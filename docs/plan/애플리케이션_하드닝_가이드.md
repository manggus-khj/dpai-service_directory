# 애플리케이션 보안 하드닝 가이드 (최소 기준)

> 문서 상태: 공통 보안 기준선
> 적용 대상: 자사 개발 Windows 데스크톱 애플리케이션 및 웹 애플리케이션 전체
> 성격: 프로젝트별 요구사항과 무관하게 **모든 제품이 최소한 충족해야 하는 공통 기준(baseline)**
> 최종 정리일: 2026-07-17
> 개별 프로젝트는 이 기준 위에 고객사 보안 요구사항(망분리, 국정원 검증 암호모듈 등)을 추가로 적용한다.

---

## 적용 판정 규칙

- 이 문서의 체크박스와 아래 요약표의 “대상”은 준수 완료 표시가 아니라 설계·구현·릴리스 검증 항목이다.
- 프로젝트는 각 항목을 `적용`, `해당 없음`, `예외 승인`, `미결정` 중 하나로 판정하고 근거를 남긴다.
- “해당 없음”은 현재 기능이나 공격 표면이 없을 때만 사용하며, 설계 변경 시 다시 평가한다.
- 기준을 완화하려면 사유, 위험, 보완 통제, 적용 기간과 승인자를 기록한다. 문서 사이의 충돌을 암묵적인 예외로 취급하지 않는다.
- 보안 방식이 미정인 기능은 평문, 무인증 또는 넓은 권한을 임시 기본값으로 삼아 완료 처리하지 않는다.

서비스 디렉토리의 프로젝트별 적용 판정은 [개발계획의 하드닝 적용](./서비스디렉토리_개발계획.md#8-프로젝트-하드닝-적용)을 따른다.

---

## 1. 기본 원칙

1. **최소 권한 (Least Privilege)** — 앱, 서비스 계정, DB 계정 모두 필요한 최소 권한만 부여한다. 관리자/DBA 권한으로 상시 동작하는 구조를 금지한다.
2. **기본값은 안전하게 (Secure by Default)** — 설치 직후 상태가 가장 안전한 상태여야 한다. 보안 기능을 사용자가 켜야 하는 구조가 아니라, 끄려면 명시적으로 꺼야 하는 구조로 만든다.
3. **심층 방어 (Defense in Depth)** — 한 겹의 방어(예: 방화벽)에 의존하지 않는다. 내부망 전용 시스템이라도 인증·암호화를 생략하지 않는다.
4. **신뢰 경계에서 모두 검증** — 클라이언트 입력, 외부 API 응답, 파일, 설정값 등 신뢰 경계를 넘어오는 모든 데이터는 서버(또는 수신 측)에서 검증한다. 클라이언트 측 검증은 UX일 뿐 보안이 아니다.
5. **실패는 안전하게 (Fail Secure)** — 오류·예외 발생 시 접근을 허용하는 방향이 아니라 차단하는 방향으로 동작해야 한다.

## 2. 공통 하드닝 항목 (Windows 앱 · 웹 앱 공통)

### 2.1 시크릿 관리

- [ ] 소스코드, 설정 파일, 리포지토리에 **비밀번호·API 키·인증서 개인키를 평문으로 저장하지 않는다.** (하드코딩 절대 금지)
- [ ] 시크릿은 환경변수, OS 자격 증명 저장소(Windows: DPAPI/Credential Manager), 또는 시크릿 관리 도구를 통해 주입한다.
- [ ] Git 히스토리에 시크릿이 커밋된 적이 있다면 해당 시크릿은 **유출된 것으로 간주하고 즉시 교체**한다.
- [ ] 개발/스테이징/운영 환경의 시크릿을 분리한다. 운영 DB 비밀번호를 개발 환경에서 재사용하지 않는다.

### 2.2 인증 (Authentication)

- [ ] 기본 계정(admin/admin 등)을 그대로 두고 출고하지 않는다. 최초 실행 시 비밀번호 변경을 강제한다.
- [ ] 비밀번호는 **bcrypt, Argon2, PBKDF2** 등 검증된 알고리즘으로 해싱한다. MD5/SHA-1/SHA-256 단독 해싱, 평문·가역 암호화 저장을 금지한다.
- [ ] 로그인 실패에 대해 **횟수 제한 또는 지연(rate limit / lockout)** 을 적용한다. (예: 5회 실패 시 잠금 또는 점증 지연)
- [ ] 로그인 실패 메시지는 "아이디 또는 비밀번호가 올바르지 않습니다"로 통일한다. 계정 존재 여부를 노출하지 않는다.
- [ ] 관리자 기능에는 가능하면 2단계 인증(MFA)을 지원한다.

### 2.3 인가 (Authorization)

- [ ] 권한 검사는 **모든 요청·기능 호출마다 서버(수신) 측에서** 수행한다. "메뉴를 숨겼으니 접근 못한다"는 인가가 아니다.
- [ ] 리소스 접근 시 소유권/권한을 확인한다. (예: `/api/users/123` 조회 시 요청자가 123에 접근 가능한지 검증 — IDOR 방지)
- [ ] 역할 기반 접근 제어(RBAC)를 기본 구조로 사용하고, 권한 상승 경로(일반 사용자 → 관리자)가 없는지 점검한다.

### 2.4 전송 구간 암호화

- [ ] 네트워크를 지나는 모든 인증 정보·개인정보·영상 메타데이터는 **TLS 1.2 이상**으로 암호화한다. TLS 1.0/1.1, SSLv3은 비활성화한다.
- [ ] 내부망 전용이라는 이유로 HTTP 평문 통신을 기본값으로 두지 않는다.
- [ ] 인증서 검증을 코드에서 우회하지 않는다. (`ServerCertificateValidationCallback => true`, `verify=False` 류의 코드 금지 — 개발 편의로 넣었다가 출고되는 대표 사례)

### 2.5 저장 데이터 보호

- [ ] 개인정보(이름, 연락처, 차량번호 등)와 인증 정보는 저장 시 암호화하거나 접근 통제를 적용한다.
- [ ] DB 접속 계정은 애플리케이션 전용 계정을 만들고 필요한 스키마·권한만 부여한다. `sa`, `root` 사용 금지.
- [ ] 백업 파일도 운영 데이터와 동일한 보호 수준(암호화, 접근 통제)을 적용한다.
- [ ] 임시 파일·로그·크래시 덤프에 비밀번호, 토큰, 개인정보가 남지 않도록 한다.

### 2.6 입력 검증 및 인젝션 방지

- [ ] DB 질의는 **파라미터 바인딩(Prepared Statement)** 만 사용한다. 문자열 연결로 SQL을 조립하지 않는다.
- [ ] OS 명령 실행이 필요한 경우 셸을 거치지 않는 API를 사용하고, 사용자 입력을 명령 문자열에 직접 넣지 않는다.
- [ ] 파일 경로 입력은 정규화 후 허용 디렉터리 내부인지 검증한다. (`../` 경로 조작 방지)
- [ ] 입력 검증은 **허용 목록(allowlist)** 방식을 우선한다. (차단 목록 방식은 우회되기 쉬움)

### 2.7 로깅 및 감사

- [ ] 로그인 성공/실패, 권한 변경, 설정 변경, 데이터 삭제 등 **보안 관련 이벤트는 반드시 로깅**한다. (누가, 언제, 무엇을, 어디서)
- [ ] 로그에 비밀번호, 토큰, 카드번호, 주민번호 등 민감정보를 남기지 않는다. 필요 시 마스킹한다.
- [ ] 로그 파일은 일반 사용자가 수정·삭제할 수 없도록 권한을 제한한다.
- [ ] 로그 보존 기간과 순환(rotation) 정책을 정의한다.

### 2.8 오류 처리

- [ ] 최종 사용자에게 스택 트레이스, DB 오류 메시지, 내부 경로, 버전 정보를 노출하지 않는다. 상세 내용은 서버 로그로만 남긴다.
- [ ] 운영 빌드에서 디버그 모드·상세 오류 페이지를 비활성화한다.

### 2.9 의존성(서드파티) 관리

- [ ] 사용 중인 라이브러리·프레임워크·런타임 목록을 관리하고, **알려진 취약점(CVE)을 주기적으로 점검**한다. (예: `dotnet list package --vulnerable`, `npm audit`, Dependabot)
- [ ] EOL(지원 종료)된 프레임워크·런타임 위에 신규 제품을 만들지 않는다.
- [ ] 패키지는 공식 저장소에서만 받고, 빌드 시 버전을 고정(lock)한다.

### 2.10 업데이트 및 패치

- [ ] 제품에 보안 패치를 배포할 수 있는 경로(업데이트 절차)를 확보한다.
- [ ] 업데이트 파일은 HTTPS로 배포하고, **해시(체크섬) 검증**을 통과한 경우에만 설치한다.

## 3. Windows 데스크톱 애플리케이션 하드닝

### 3.1 실행 권한

- [ ] 매니페스트의 요청 권한은 기본적으로 `asInvoker`로 한다. 관리자 권한(`requireAdministrator`)은 실제로 필요한 기능이 있을 때만, 가능하면 해당 기능만 분리하여 요구한다.
- [ ] Windows 서비스는 `LocalSystem`이 아니라 필요 권한에 맞는 계정(`LocalService`, `NetworkService`, 전용 서비스 계정)으로 실행한다.

### 3.2 설치 및 파일 권한

- [ ] 실행 파일은 `Program Files` 아래에 설치한다. (일반 사용자가 실행 파일을 교체할 수 없는 위치)
- [ ] `C:\MyApp` 같은 루트 경로 설치나 Everyone 쓰기 권한이 걸린 설치 폴더를 만들지 않는다. — 일반 사용자가 exe/dll을 바꿔치기하면 **권한 상승 취약점**이 된다.
- [ ] 사용자별 데이터·설정은 `%ProgramData%` 또는 `%AppData%`에 저장하고, 폴더 ACL을 용도에 맞게 제한한다.

### 3.3 DLL 하이재킹 방지

- [ ] DLL은 전체 경로로 로드하거나 `SetDefaultDllDirectories` / `LoadLibraryEx(LOAD_LIBRARY_SEARCH_SYSTEM32)` 를 사용해 검색 경로를 제한한다.
- [ ] 쓰기 가능한 디렉터리(다운로드 폴더, 임시 폴더)에서 실행되는 설치 프로그램은 특히 DLL 검색 경로 하이재킹에 취약하므로 점검한다.

### 3.4 로컬 시크릿 저장

- [ ] 로컬에 저장해야 하는 자격 증명(DB 접속 정보, API 토큰 등)은 **DPAPI(`ProtectedData`) 또는 Windows Credential Manager**로 보호한다. 설정 파일 평문 저장 금지.
- [ ] "암호화했다"며 코드에 고정 키를 박아 XOR/AES 하는 방식은 평문과 동급으로 간주한다.

### 3.5 로컬 통신 (IPC)

- [ ] Named Pipe, 로컬 소켓, 로컬 HTTP 서버 등 프로세스 간 통신 채널에도 **접근 제어(ACL) 또는 인증**을 적용한다. `localhost`라는 이유로 무인증으로 두지 않는다. (같은 PC의 다른 사용자·악성코드도 localhost에 접근 가능)
- [ ] 로컬 웹 서버를 내장하는 경우 바인딩을 `127.0.0.1`로 제한한다. `0.0.0.0` 바인딩 금지.

### 3.6 빌드 보호 기능

- [ ] 컴파일 시 OS 보호 기능을 활성화한다: **DEP/NX, ASLR(동적 기준 주소), CFG(Control Flow Guard)**. (.NET은 기본 적용, 네이티브/C++ 모듈은 명시적으로 확인)
- [ ] 운영 빌드에 디버그 심볼·테스트 코드·개발자 백도어(숨겨진 단축키, 인증 우회 플래그)를 포함하지 않는다.

### 3.7 자동 업데이트

- [ ] 업데이트 확인·다운로드는 HTTPS로만 수행한다.
- [ ] 다운로드된 업데이트는 서버에 게시된 **해시(체크섬)와 대조하여 무결성 검증 후** 설치한다. 검증 없는 자동 업데이트는 원격 코드 실행 통로가 된다. 해시 파일 역시 HTTPS로만 받는다.

## 4. 웹 애플리케이션 하드닝

### 4.1 HTTPS 및 보안 헤더

- [ ] 전 구간 HTTPS를 강제하고 HTTP 요청은 HTTPS로 리다이렉트한다. **HSTS** 헤더를 적용한다.
- [ ] 최소 보안 헤더를 적용한다:
  - `Content-Security-Policy` (최소한 `frame-ancestors`, 가능하면 script 소스 제한)
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY` (또는 CSP frame-ancestors로 대체)
  - `Referrer-Policy: strict-origin-when-cross-origin`
- [ ] 서버·프레임워크 버전 노출 헤더(`Server`, `X-Powered-By`)를 제거하거나 최소화한다.

### 4.2 세션 및 쿠키

- [ ] 세션 쿠키에 **`HttpOnly`, `Secure`, `SameSite=Lax` 이상**을 설정한다.
- [ ] 로그인 성공 시 세션 ID를 재발급한다. (세션 고정 공격 방지)
- [ ] 유휴 타임아웃과 절대 타임아웃을 설정하고, 로그아웃 시 서버 측 세션을 실제로 무효화한다.
- [ ] JWT 사용 시: 서명 알고리즘을 서버에서 고정(`none` 금지), 만료 시간을 짧게 설정, 토큰을 localStorage보다는 HttpOnly 쿠키에 저장하는 것을 우선 검토한다.

### 4.3 XSS / CSRF

- [ ] 출력 시점에 컨텍스트에 맞는 이스케이프/인코딩을 적용한다. 템플릿 엔진의 자동 이스케이프를 끄지 않는다. (`dangerouslySetInnerHTML`, `v-html`, `Html.Raw` 사용은 리뷰 필수)
- [ ] 상태를 변경하는 모든 요청(POST/PUT/DELETE)에 **CSRF 토큰**을 적용한다. (SameSite 쿠키는 보조 수단이지 대체재가 아니다)

### 4.4 API 보안

- [ ] 모든 API 엔드포인트는 인증·인가를 거친다. "프론트에서만 호출하는 API"도 예외가 아니다.
- [ ] CORS는 허용 오리진을 명시적으로 지정한다. `Access-Control-Allow-Origin: *` 와 인증을 함께 쓰지 않는다.
- [ ] 요청 크기 제한, 페이지네이션, 속도 제한(rate limit)을 적용한다.
- [ ] API 응답에 필요한 필드만 내려보낸다. 엔티티 전체 직렬화 금지. (Mass assignment / 과다 노출 방지)

### 4.5 파일 업로드

- [ ] 확장자와 MIME 타입을 **허용 목록**으로 검증하고, 파일 내용(매직 바이트)도 확인한다.
- [ ] 업로드 파일은 웹 루트 밖 또는 실행이 차단된 저장소에 저장하고, 파일명은 서버가 재생성한다. (원본 파일명 그대로 저장 금지)
- [ ] 파일 크기 제한을 적용한다.

### 4.6 서버 및 인프라 설정

- [ ] 디렉터리 리스팅을 비활성화한다.
- [ ] 관리 콘솔·관리자 페이지는 접근 IP를 제한하거나 별도 인증 계층을 둔다.
- [ ] `.git`, `.env`, 백업 파일(`*.bak`, `*.old`), 설정 파일이 웹에서 접근되지 않는지 확인한다.
- [ ] 운영 서버에서 불필요한 포트·서비스·샘플 페이지·기본 문서를 제거한다.
- [ ] DB, 캐시(Redis 등), 내부 API는 외부에서 직접 접근할 수 없도록 네트워크를 분리한다.

## 5. 개발·배포 프로세스

- [ ] **코드 리뷰**: 인증·인가·암호화·입력 처리 관련 변경은 리뷰 없이 병합하지 않는다.
- [ ] **정적 분석(SAST)** 및 시크릿 스캔을 CI에 포함한다. (예: gitleaks, CodeQL, SonarQube)
- [ ] **의존성 취약점 스캔**을 CI 또는 정기 작업으로 수행한다.
- [ ] 빌드는 개발자 PC가 아닌 **빌드 서버에서 재현 가능하게** 수행하고, 배포 산출물의 해시를 기록해 둔다.
- [ ] 운영 배포 계정·서버 접근 권한을 최소 인원으로 제한하고 접근 기록을 남긴다.
- [ ] 취약점 신고 접수 및 대응 절차(누가, 며칠 내 패치)를 정의해 둔다.
