# 서비스 디렉토리 인증서 전환 변경계획

> 문서 상태: 목표 설계 확정, 코드·XSD·설치 소스 미반영
> 기준일: 2026-07-19
> 적용 기준: 사내 `Directory서비스_애플리케이션_하드닝_가이드` 개정본

이 문서는 현재 구현된 평문 HTTP·승인 대기 기반 서비스 디렉토리를 사이트 CA·HTTPS·즉시 등록 모드 기반으로 전환하기 위한 변경 범위와 구현 순서를 정의한다. 요청·응답의 목표 wire 계약은 [외부 애플리케이션 API 명세](./서비스디렉토리_외부애플리케이션_API명세.md)와 [내부 API 명세](./서비스디렉토리_내부_API명세.md)가 단일 원본이다.

문서 확정은 구현 완료를 뜻하지 않는다. 현재 코드는 HTTP listener, 일일 API 키, `pending.xml`, 승인·거절 Admin API와 승인 대기 UI를 사용한다. 이 계획의 HTTPS·CA·CRL·등록 모드·인증서 발급 계약은 후속 코드 변경 전까지 미구현이다.

## 1. 확정 결정

| 항목 | 목표 결정 |
|---|---|
| Directory 주소 발견 | 외부 앱이 성공한 Milestone Management Server session에서 Directory가 함께 설치된 서버의 DNS hostname/FQDN 한 개와 실제 remote IPv4 한 개를 각각 `DirectoryHostName`, `DirectoryIpv4Address`로 얻는다. 둘 중 선택한 주소로 Directory에 접속하되 둘 다 Directory TLS identity로 저장한다. 별도 연결정보 파일과 Directory 주소 입력을 사용하지 않는다. |
| 등록 서비스 주소 | 등록 외부 앱이 자기 서비스의 DNS hostname/FQDN 한 개와 서비스가 bind할 IPv4 한 개를 각각 `ServiceHostName`, `ServiceIpv4Address`로 선택·영속화해 요청한다. Milestone/Directory 주소, TCP source IP와 DNS 역조회 값으로 대체하지 않는다. |
| 주소 지원 범위 | Directory·External service·Peer·installer·방화벽·인증서 IP SAN은 IPv4만 지원한다. IPv6 입력·listener·endpoint·SAN은 거부한다. |
| 최초 신뢰 | 해당 Milestone 서버 identity별로 Directory 사이트 CA 인증서와 SHA-256 SPKI pin을 TOFU로 저장한다. 최초에도 SAN·chain·CA 제약·키 강도를 검증하고, 이후 pin 불일치는 같은 주소라도 fail closed한다. |
| Directory CA | 최초 설치 때 사이트 CA와 Directory HTTPS leaf를 자동 생성한다. CA 인증서에는 endpoint SAN을 넣지 않고 Directory leaf에만 `DirectoryHostName` DNS SAN과 `DirectoryIpv4Address` IP SAN을 모두 넣는다. CA 최대 유효기간은 20년이며 개인키는 제한 ACL과 OS 보호를 적용한다. 암호화 CA 백업·복원·key rotation·dual-pin 절차를 제공한다. |
| 원격 전송 | External·Peer 원격 API는 TLS 1.2 이상을 지원하는 OS 보안 기본값의 HTTPS만 허용한다. protocol·cipher suite를 앱 코드에 고정하지 않고 구버전 비활성화를 OS별 통합 검증한다. 평문 HTTP listener와 redirect fallback을 제공하지 않는다. |
| 로컬 전송 | Admin과 와치독 loopback은 로컬 IPC 경계로 분류해 기존 `127.0.0.1` 제한과 Negotiate·endpoint 검증을 유지한다. 원격 External·Peer HTTPS와 경로·prefix를 분리한다. |
| External 요청 검증 | 기존 44자 일일 API 키 알고리즘을 유지하고 health·조회·등록의 ProductCode 검증 및 요청 admission에 사용한다. 강한 호출자 identity나 요청 서명으로 표현하지 않는다. |
| 등록 승인 | 승인 대기 요청을 만들지 않는다. 인증된 관리자가 설정 UI의 등록 서비스 화면에서 등록 모드를 연 행위가 1회성 사전 승인이다. |
| 등록 모드 | ProductCode를 입력하거나 선택하지 않는 전역 창이다. 최대 1시간 동안 첫 번째 유효 등록 요청 한 건만 허용하고 성공·수동 종료·만료·서비스 재시작·불확실한 장애 시 닫는다. |
| 설치 입력 | 설치하는 사람에게 ProductCode, Directory 주소, CA 파일, pin 또는 인증서/PFX를 요구하지 않는다. |
| 인증서 발급 | 열린 등록 모드에서 첫 번째 유효 `POST /api/registration`이 서비스 등록과 서버 인증서 발급을 같은 복구 단위로 완료하고 즉시 결과를 반환한다. 등록 서비스 leaf에는 요청한 `ServiceHostName` DNS SAN과 `ServiceIpv4Address` IP SAN을 모두 넣는다. 개인키는 앱 서버에서 생성하며 Directory로 전송하지 않는다. |
| 재등록 | 등록 모드가 열린 상태의 새 CSR은 같은 ProductCode의 기존 인증서를 폐기하고 새 등록·인증서로 교체할 수 있다. 정확한 idempotent 재시도는 등록 모드가 닫힌 뒤에도 기존 결과를 반환한다. |
| 갱신 | 현재 유효하고 폐기되지 않은 인증서 개인키의 요청 서명과 새 CSR proof-of-possession으로 자동 갱신한다. 저장한 서비스 hostname/FQDN 또는 IPv4를 변경하면 변경 뒤의 완전한 두 값을 service identity hash에 결합해 재발급한다. ProductCode 변경, 인증서 분실·폐기·만료는 등록 모드를 다시 열어 재등록한다. |
| 폐기 | 서비스 삭제·재등록·운영자 폐기에서 serial ledger와 CRL을 갱신한다. 서비스 목록 삭제만으로 인증서를 유효하게 남기지 않는다. |
| UI | 트레이 context menu에는 등록 모드를 넣지 않는다. 설정 UI의 `등록 서비스` 화면에 상태·남은 시간·시작·종료를 배치하고 `승인 대기` 메뉴·화면을 제거한다. |

## 2. 보안 모델

### 2.1 신뢰 방향

서비스 디렉토리는 서로 바꿀 수 없는 두 network identity와 다음 다섯 신뢰 동작을 분리한다.

1. 외부 앱은 이미 성립한 Milestone Management Server session에서 얻은 `DirectoryHostName`·`DirectoryIpv4Address` 쌍을 Directory 위치로 신뢰한다.
2. 최초 Directory TLS 연결에서는 Directory leaf의 DNS·IPv4 SAN이 그 쌍과 모두 일치하는지 포함한 엄격한 TOFU 검증 뒤 사이트 CA와 SPKI pin을 저장한다.
3. 등록 서버 앱은 Directory 위치와 무관한 자기 서비스의 `ServiceHostName`·`ServiceIpv4Address` 쌍을 선택·영속화하고 CSR과 등록 요청에 함께 넣는다.
4. Directory는 열린 등록 모드와 유효한 일일 API 키·CSR을 결합해 한 건의 발급을 허용하고, 서비스 leaf의 DNS·IP SAN을 그 서비스 쌍으로 구성한다.
5. 외부 클라이언트는 조회한 서비스에 직접 연결할 때 같은 사이트 CA·pin·SAN·CRL로 대상 서버 인증서를 검증한다.

신뢰한 주소는 인증서 검증을 생략하는 근거가 아니다. 최초 연결에서도 접속 host와 SAN 일치, leaf→사이트 CA 서명, CA `BasicConstraints`, EKU·Key Usage, 유효기간과 알고리즘 강도를 검증한다. 최초 pin 저장 뒤에는 CA가 바뀌어도 자동 재신뢰하지 않는다.

Directory 전용 가이드의 우선 bootstrap인 접속정보 파일은 이 제품에서 생성·배포할 수 없다. 대신 이미 성공한 Milestone session의 Directory hostname/FQDN·IPv4 쌍을 위치 신뢰 근거로 사용하고 가이드가 허용한 TOFU fallback만 적용한다. 이 쌍은 Directory 접속과 Directory leaf 검증 전용이며 등록 서비스 record나 등록 서비스 leaf SAN에 복사하지 않는다. 외부 앱의 신뢰 초기화에는 로컬 관리자 권한과 명시적 확인을 요구하며, 외부 앱 제거 시 저장 CA·pin·SiteId·CRL cache를 완전히 삭제한다. Directory 제품의 일반 제거가 운영 CA를 보존하는 정책과 외부 앱의 local trust 삭제 정책은 서로 다른 수명주기다.

### 2.2 일일 API 키 예외

일일 API 키는 ProductCode와 로컬 날짜에서 생성되므로 알고리즘을 아는 주체를 암호학적으로 구분하지 못한다. method·path·CSR·body에 결합되지 않고 같은 날짜 replay도 막지 못한다. 사내 문서 배포 제한과 구현 비공개는 보완 통제일 수 있지만 암호학적 비밀로 간주하지 않는다.

새 하드닝 가이드가 요구하는 강한 초기 등록 credential 대신 다음 결합을 발급 승인 경계로 사용하는 프로젝트 예외를 기록한다.

- 로컬 Negotiate와 운영자 그룹 인가를 통과한 관리자가 직접 등록 모드를 연다.
- 등록 창은 ProductCode 입력 없이 전역으로 열리지만 1시간·유효 요청 1건으로 제한한다.
- External 요청은 HTTPS, 일일 API 키와 ProductCode 일치, exact local endpoint, rate limit을 통과해야 한다.
- CSR 자체 서명·키 강도와 requested SAN이 요청의 `ServiceHostName` DNS 한 개·`ServiceIpv4Address` IPv4 한 개와 정확히 같은지 검증한다. Directory는 TCP source IP나 DNS 역조회 결과로 두 값을 만들거나 교정하지 않는다.
- 첫 유효 요청만 원자적으로 claim하고 성공 즉시 창을 닫는다.
- 발급·재발급·폐기·등록 모드 전이를 보안 감사 로그에 남긴다.

이 예외의 잔여 위험은 열린 시간에 알고리즘과 ProductCode를 아는 다른 주체가 먼저 유효 요청을 보내는 first-wins 경쟁이다. 운영자는 등록 모드 화면의 결과와 등록 서비스·인증서 정보를 확인하고 잘못된 항목을 즉시 삭제·폐기한 뒤 창을 다시 연다. 삭제 뒤 클라이언트 CRL 갱신 전까지의 폐기 전파 지연도 잔여 위험으로 기록한다.

## 3. 등록 모드 상태 머신

```text
CLOSED
  │ POST /admin/registration-mode/open
  ▼
OPEN ── 1시간 만료·수동 종료·서비스 종료 ──▶ CLOSED
  │ 첫 번째 완전히 검증된 등록 요청의 원자 claim
  ▼
CLAIMED
  ├─ 등록·발급 commit 성공 ──────────────▶ CLOSED
  └─ 결과 불확실·저장/서명 실패 ─────────▶ CLOSED
```

규칙:

- 등록 모드는 process memory에만 존재하고 `config.xml`에 영속화하지 않는다.
- `OPEN`은 UTC 만료 시각과 monotonic elapsed time을 함께 사용한다. 벽시계 역행으로 1시간을 넘기지 않는다.
- 이미 `OPEN`일 때 open 요청은 현재 창을 반환하고 만료를 연장하지 않는다.
- XML·일일 키·CSR·도메인 검증에 실패한 요청은 창을 소비하지 않지만 rate limit과 보안 감사를 적용한다.
- 검증을 모두 통과한 뒤 공용 mutation gate에서 `OPEN -> CLAIMED`를 원자적으로 수행한다.
- `CLAIMED` 뒤에는 두 번째 요청을 받지 않는다. 발급 결과가 불확실하면 안전하게 닫고 운영자가 상태를 확인한 뒤 다시 연다.
- 성공한 요청의 `RegistrationRequestId`·CSR SHA-256·정규화 payload·발급 serial을 내구적으로 보존한다. 정확히 같은 재시도는 창이 닫혀 있어도 같은 공개 인증서 결과를 반환한다.
- 같은 request ID에 다른 CSR 또는 payload가 오면 `CONFLICT`다.

## 4. 인증서 수명주기

### 4.1 사이트 CA와 Directory leaf

- 기본 모드는 Directory가 로컬에서 생성한 site root CA다.
- CA 인증서는 `BasicConstraints CA=TRUE, pathLen=0`, `keyCertSign`, `cRLSign`, SHA-256 이상을 사용한다.
- CA 인증서는 issuer 신뢰 anchor이므로 endpoint hostname·IP SAN을 갖지 않는다.
- Directory leaf는 `Server Authentication`, `digitalSignature`, `DirectoryHostName` DNS SAN 한 개와 `DirectoryIpv4Address` IP SAN 한 개를 모두 가진다.
- 서버 앱 leaf는 `Server Authentication`, `digitalSignature`, exact `ServiceHostName` DNS SAN 한 개·`ServiceIpv4Address` IPv4 SAN 한 개, RSA 2048 이상 또는 ECDSA P-256 이상을 사용하며 기본 유효기간은 1년이다.
- Directory가 반환하거나 TLS chain에 포함하는 root는 공개 인증서뿐이며 CA 개인키는 절대 API로 노출하지 않는다.

### 4.2 CSR과 SAN

- 앱 서버가 키pair와 PKCS#10 CSR을 로컬에서 생성한다.
- 등록 서버 앱은 최초 설정에서 서비스가 사용할 DNS hostname/FQDN 한 개와 서비스 listener가 bind할 IPv4 한 개를 사용자가 선택하게 하고 제한 ACL 설정에 한 쌍으로 영속화한다.
- 앱은 요청 직전에 선택한 IPv4가 활성 서비스 NIC에 할당되고 listener가 사용할 수 있는지, 선택한 hostname/FQDN의 정방향 DNS가 그 IPv4를 포함하는지 확인한다. 매 요청마다 NIC 전체를 열거해 등록 identity를 바꾸지 않는다.
- `ServiceHostName`은 canonical lowercase ASCII DNS hostname/FQDN 한 개, `ServiceIpv4Address`는 canonical dotted-decimal IPv4 한 개다. loopback, wildcard, APIPA, multicast, broadcast, 선행 0과 모든 IPv6 표현을 거부한다.
- CSR requested SAN은 정확히 `dNSName=ServiceHostName` 한 개와 `iPAddress=ServiceIpv4Address` 한 개만 포함한다. Directory는 CSR 자체 서명, 공개키 강도, 허용 extension과 요청·CSR의 두 값 일치를 확인해 동일한 두 SAN만 발급한다.
- Milestone/Directory의 hostname·IPv4, 요청 TCP source IP, DNS 역조회와 Directory 관점의 routing 주소를 등록 service identity나 SAN으로 사용하지 않는다. Directory가 원격 주소 소유권을 암호학적으로 증명한다고 표현하지 않는다.

### 4.3 갱신·재등록·폐기

- 동일 ProductCode의 갱신은 현재 leaf 개인키로 ProductCode·새 CSR hash·정규화한 `ServiceHostName`·`ServiceIpv4Address` identity hash를 포함한 canonical 요청을 서명하고 새 CSR 자체 서명을 검증한 뒤 자동 처리한다.
- 서버 앱은 만료 30일 전뿐 아니라 명시적 설정 변경으로 저장한 hostname/FQDN 또는 IPv4가 바뀌면 즉시 갱신 endpoint로 재발급을 요청한다. 실제 변경값이 한쪽뿐이어도 변경 뒤의 hostname·IPv4 두 필드를 항상 완전한 쌍으로 제출해야 하며 필드 하나를 생략한 부분 갱신은 거부한다. 자동 NIC 열거로 identity를 바꾸지 않는다. ProductCode 변경, private key 분실, 폐기·만료 leaf만 등록 모드를 다시 요구한다.
- 재등록 commit은 이전 active serial 폐기, 새 인증서 발급, 서비스 레코드 교체와 CRL 갱신을 하나의 복구 가능한 작업으로 수행한다.
- 삭제도 서비스 톰스톤과 모든 active serial 폐기·CRL 갱신이 함께 성공해야 완료다.
- CRL은 표준 X.509 DER, SHA-256 이상 서명, 단조 증가 `CRLNumber`, `thisUpdate`·`nextUpdate`를 사용한다.
- 클라이언트는 유효한 CRL을 얻지 못하면 이전에 같은 endpoint에서 검증한 정확히 같은 leaf만 제한적으로 재연결하고 새·변경 인증서를 거부한다.

### 4.4 CA 갱신과 교체

- 같은 CA key로 인증서만 갱신하면 SPKI pin은 유지한다.
- CA key rotation은 현재 정상 TLS 연결을 통해 다음 CA와 pin을 전달하고 dual-pin 기간을 둔다.
- 전환을 놓친 앱과 예고 없는 CA 변경은 자동 신뢰하지 않는다. 해당 Milestone server identity의 Directory 신뢰 초기화 후 TOFU를 다시 수행한다.
- 일반 uninstall은 CA·ledger·CRL·백업을 보존하고 명시적 전체 삭제에서만 제거한다.

## 5. 데이터와 복구 변경

| 대상 | 변경 |
|---|---|
| `pending.xml` | 목표 설계에서 제거한다. 기존 구현 데이터는 코드 전환 시 명시적 migration 또는 미출시 초기화 정책으로 처리하고 조용히 버리지 않는다. |
| `directory.xml` | 각 서비스 레코드에 `ServiceHostName`·`ServiceIpv4Address`를 필수 쌍으로 저장하고 현재 인증서 serial·SPKI fingerprint·동일한 두 SAN·유효기간을 연계하거나 별도 ledger identity를 참조한다. 외부 조회에는 내부 ledger 정보를 노출하지 않는다. |
| `config.xml` | CA mode, SiteId, `DirectoryHostName`·`DirectoryIpv4Address`, CRL publish 설정과 CA rotation 상태를 새 schema migration으로 추가한다. ProductCode와 등록 모드 상태는 저장하지 않는다. |
| CA private key | OS key store 또는 DPAPI와 서비스 SID ACL로 보호한다. 정확한 provider·key export 정책은 .NET Framework 4.8·지원 OS에서 검증 후 확정한다. |
| certificate ledger | CSPRNG 16바이트 positive unique serial, ProductCode, request ID, CSR/SPKI hash, SAN, issue/expiry/revoke 시각·사유, idempotent 응답 근거와 CRL high-water를 영속화한다. |
| CRL | 서명된 canonical DER과 CRL number high-water를 원자 교체하고 CA ledger transaction과 복구 일관성을 유지한다. |
| CA backup | 운영자가 지정한 암호로 암호화한 승인 백업만 허용하고 평문 export를 금지한다. backup 완료를 설치 완료 조건으로 취급한다. |

등록·발급 commit은 최소한 `directory`, certificate ledger, leaf artifact와 필요한 CRL 변경을 하나의 journal transaction으로 처리한다. CA 서명 뒤 저장 실패가 발생해 같은 serial을 재사용하거나 발급 사실을 잃지 않도록 serial 예약·서명·commit 순서를 별도 설계하고 fault-injection으로 검증한다.

## 6. UI와 Admin 변경

설정 UI의 좌측 메뉴는 `대시보드`, `등록 서비스`, `동기화`, `설정`으로 구성한다. `승인 대기` 메뉴와 승인·거절 화면은 제거한다.

`등록 서비스` 화면 상단에 다음 등록 모드 패널을 둔다.

- 현재 상태: `닫힘`, `등록 가능`, `등록 처리 중`
- 등록 가능 상태의 남은 시간 `HH:mm:ss`
- `등록 모드 시작` 버튼: 고정 1시간 창 시작
- `등록 모드 종료` 버튼
- 전역 창이며 첫 유효 ProductCode 한 건이 등록된다는 경고
- 마지막 등록 결과: 시각, ProductCode, 서비스 hostname/FQDN, 서비스 IPv4, 인증서 serial·만료, 성공/실패
- 등록 서비스 목록의 인증서 상태·만료와 삭제 시 폐기 경고

트레이 context menu에는 등록 모드 시작·종료를 추가하지 않는다. 설정 UI는 loopback Admin API만 사용하고 등록 모드 상태를 로컬 파일에서 읽지 않는다.

## 7. API 변경 요약

### External·PKI

- 원격 base를 같은 Directory identity에 결합한 `https://{DirectoryHostName}:21000` 또는 `https://{DirectoryIpv4Address}:21000`으로 변경
- `GET /pki/ca`: site identity·CA 공개 인증서·SPKI pin·CRL 위치
- `GET /pki/crl`: 서명된 DER CRL
- `POST /api/registration`: Directory 주소와 별개인 `ServiceHostName`·`ServiceIpv4Address`, CSR와 `RegistrationRequestId`를 포함하고 열린 창에서 즉시 등록·발급
- `POST /api/certificates/renew`: 현재 인증서 개인키 proof와 새 CSR로 자동 갱신
- `GET /api/services`: 승인 대기 의미를 제거하고 활성 등록만 반환

### Admin

- 제거: `GET /admin/pending`
- 제거: `POST /admin/pending/{id}/approve`
- 제거: `POST /admin/pending/{id}/reject`
- 추가: `GET /admin/registration-mode`
- 추가: `POST /admin/registration-mode/open`
- 추가: `POST /admin/registration-mode/close`
- 확장: 서비스 삭제는 인증서 폐기·CRL 갱신 결과를 포함
- 추가 검토: 인증서 ledger 조회·명시적 serial 폐기·CA backup/restore/rotation 운영 endpoint

### Peer

- 모든 원격 `/api/sync/*` endpoint를 HTTPS로 전환하고 기존 ECDH·SAS·HMAC을 TLS 위에 유지한다.
- 같은 사이트의 피어가 동일 CA를 사용하도록 backup/restore와 issuer 역할을 정의한다.
- active-active CA 발급은 serial·CRL split-brain 해결 전 금지하고, 우선 단일 active issuer·명시적 승격을 사용한다.

## 9. 필수 보안·장애 검증

- 최초 TOFU의 SAN 불일치, 잘못된 chain·CA 제약·알고리즘·유효기간 거부
- 저장 pin 일치와 같은 주소의 예고 없는 CA key 변경 fail-closed
- 등록 모드 닫힘, 정확히 1시간 만료, 벽시계 역행, 수동 종료와 서비스 재시작 닫힘
- ProductCode 입력 없이 서로 다른 ProductCode의 동시 유효 요청 중 정확히 한 건만 claim
- invalid 일일 키·CSR·SAN 요청이 창을 소비하지 않으며 flood가 제한되는지
- claim 직전·serial 예약·서명·ledger·directory·CRL·응답 직전 강제 종료
- 응답 유실 뒤 같은 request ID·CSR exact retry와 다른 payload 충돌
- 잘못된 등록 삭제에서 directory tombstone·serial revoke·CRL publish 원자성
- CRL 서명·number rollback·만료·cache fallback과 새 인증서 차단
- 같은 SAN 정기 갱신, 명시적으로 저장한 service hostname/FQDN·IPv4 쌍 변경 재발급, 키 rollover와 만료·폐기 인증서 갱신 거부
- Directory hostname/FQDN·IPv4 쌍이 등록 service 쌍으로 복사되지 않고 TCP source IP·DNS 역조회가 service identity를 바꾸지 않는지
- 등록 service hostname/FQDN 또는 IPv4 중 한쪽만 실제로 바뀐 경우에도 변경 뒤의 두 필드를 모두 제출한 CSR·identity hash는 허용하고 필드 하나를 생략한 부분 갱신은 거부하는지
- Directory listener·Peer endpoint·등록 service 주소와 certificate IP SAN에서 IPv6를 모두 거부하는지
- CA same-key 갱신, key rotation dual-pin, 놓친 클라이언트 trust reset
- CA backup 암호화·복원, 일반 uninstall 보존과 명시적 전체 삭제
- Peer 동일 CA, 단일 issuer와 split-brain 발급 차단
- 설치 과정에서 ProductCode·Directory 주소·CA 파일·pin·PFX 입력을 요구하지 않는지
- 설정 UI 등록 서비스 화면에서만 등록 모드를 조작하고 트레이 menu·승인 대기 화면에 기능이 없는지

## 10. 현재 구현과 후속 변경 대상

현재 소스에서 확인된 직접 변경 대상은 다음과 같다.

- `ServiceDirectoryListenerAddress`·`ServiceDirectoryHttpListenerHost`: `http://` remote prefix와 anonymous remote transport
- installer setup/state scripts: HTTP URL ACL만 구성하고 `sslcert` binding·CA 수명주기가 없음
- `DailyApiKeyCodec`·External handler: CSR·등록 모드·인증서 결과가 없는 승인 대기 계약
- Domain·External·Peer DTO: 단일 `ServerAddress`를 `ServiceHostName`·`ServiceIpv4Address` 필수 pair로 바꾸고 Directory 접속용 `DirectoryHostName`·`DirectoryIpv4Address`와 분리해야 함
- Domain/Application state: `pending.xml`과 approve/reject 상태 머신
- Admin protocol/handler/Tray: pending 3 endpoint와 승인 대기 화면
- `config.xml` v1·recovery journal: CA·ledger·CRL target과 schema가 없음
- Peer endpoint parser·outbound client: `http://` canonical endpoint 고정
- tests: HTTP·pending 계약에 고정된 기존 테스트를 목표 계약으로 교체·확장해야 함

후속 코드 변경은 위 순서대로 작은 검증 단위로 수행한다. 문서만 바뀐 현재 상태에서 기존 HTTP 구현을 새 계약 준수로 표시하지 않는다.
