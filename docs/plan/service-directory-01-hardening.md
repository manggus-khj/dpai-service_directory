## 적용 판정 규칙

- 이 문서의 체크박스와 요약표의 "대상"은 준수 완료 표시가 아니라 설계·구현·릴리스 검증 항목이다.
- 프로젝트는 각 항목을 `적용`, `해당 없음`, `예외 승인`, `미결정` 중 하나로 판정하고 근거를 남긴다.
- 이 문서는 **Directory 구조를 사용하는 제품에만** 적용한다. 네트워크 기능이 없는 단독 프로그램, Directory를 사용하지 않는 제품은 이 문서 전체가 "해당 없음"이며, 공통 하드닝 가이드만 적용한다.
- 기준을 완화하려면 사유, 위험, 보완 통제, 적용 기간과 승인자를 기록한다. 이 문서가 명시적으로 허용한 잔여 위험 수용 항목(4.3 TOFU, 5절 자동 재신뢰)도 프로젝트 적용표에 기록한다.

서비스 디렉토리의 프로젝트별 적용 판정은 [개발계획의 하드닝 적용](./service-directory-03-development.md#8-프로젝트-하드닝-적용)을 따른다.

## 1. 전제 구조

```
[클라이언트] ──(1) 서버 주소 조회──▶ [Directory 서비스] ◀──(2) 등록/발급── [서버 서비스 A, B, ...]
     │                                    (= 사이트 CA)
     └──(3) 직접 TLS 통신 (핀 검증)──▶ [서버 서비스]
```

- 현장(사이트)당 신뢰 기준점(trust anchor)은 **Directory CA 하나**다. 클라이언트에 서버별 인증서·핀을 개별 배포하지 않는다.
- 폐쇄망 배포를 기본 전제로 하며, 공인 CA 인증서·인터넷 접근을 요구하지 않는다.

## 2. 전송 구간 암호화 원칙

- [ ] **TLS 1.2 이상을 지원하는 OS 및 런타임을 사용하고, TLS 버전과 Cipher Suite 선택은 원칙적으로 OS의 보안 기본값에 위임한다.** (OS 업데이트를 통해 TLS 1.3 및 향후 정책 변경이 자연 반영되도록) SSLv3, TLS 1.0/1.1이 활성화되지 않았는지 배포 대상 OS 정책과 통합 테스트로 검증한다. 특정 프로토콜·suite 고정은 고객 요구나 호환성 사유가 있는 경우에만 예외 승인으로 적용한다.
- [ ] 내부망 전용이라는 이유로 HTTP 평문 통신을 기본값으로 두지 않는다.
- [ ] 인증서 검증을 코드에서 우회하지 않는다. (`ServerCertificateValidationCallback => true`, `verify=False` 류의 "어떤 인증서든 통과" 코드 금지) **핀 기반 검증 또는 Directory CA 체인 검증으로 대체하는 것은 우회가 아니다.**
- [ ] **인터넷상의 OCSP/CRL 엔드포인트에 의존하지 않는다.** (외부 조회 시도는 폐쇄망에서 접속 지연·실패의 원인) 인증서 폐기 검증은 4.4의 내부 CRL로 수행한다.

## 3. Directory CA — 발급 체계

### 3.1 Directory (사이트 CA)

- [ ] Directory는 설치(최초 기동) 시 **CA 키쌍과 루트 인증서를 자동 생성**한다. CA 유효기간은 **최대 20년**으로 한다.
  - *운영 원칙: **CA 인증서 갱신과 CA 키 교체를 구분한다.** 기존 키를 유지한 인증서 갱신은 SPKI가 변하지 않으므로 클라이언트 핀 교체가 필요 없다(만료 접근 시 동일 키로 갱신하고 새 CA 인증서를 기존 TLS 세션으로 배포). 핀 롤오버(4.5)가 필요한 것은 **키 교체 시뿐**이며, 키 교체는 현장 전체 재신뢰 이벤트이므로 키 유출·알고리즘 지원 종료·암호 정책 변경 등 필요한 경우에만 수행한다. 20년은 보장 사용 기간이 아니라 최대치이며, 조기 키 교체 절차(4.5 롤오버 + 5절 복구)는 만료와 무관하게 항상 보유한다. 양자내성 암호 전환은 OS·런타임 지원 시점에 본 정책을 재검토한다.*
- [ ] CA 개인키는 소프트웨어 키 저장소에 보관하되, **일반 사용자·일반 관리 도구를 통한 개인키 내보내기를 허용하지 않으며, 승인된 암호화 CA 백업 절차에서만 제한적으로 내보낼 수 있다.** Directory 서비스 계정에만 접근을 허용하는 ACL을 적용하고, 백업 생성·복원(가져오기)에는 접근 통제와 감사 로그를 적용한다. 파일 평문 저장 금지. (TPM/HSM 고정 저장은 백업·이중화 복원과 양립하지 않으므로 본 표준에서는 채택하지 않는다)
- [ ] Directory 자신의 TLS 서비스 인증서도 자체 CA로 발급하여 사용한다.
- [ ] 인증서 발급(CSR 서명)은 **인증된 서버 등록 절차를 통과한 요청에 대해서만** 수행한다. 무인증 발급 엔드포인트를 두지 않는다. (등록 인증이 곧 CA의 보안이다)
- [ ] Directory 이중화(페일오버) 구성 시 CA 키를 백업·복원 절차에 포함하여 **양쪽이 동일한 CA를 사용**하도록 한다. (핀 불일치 방지)
- [ ] **재설치·이전 시에는 CA 백업 복원이 표준 절차다.** 설치 프로그램은 CA 백업 파일이 있으면 신규 생성 대신 복원을 우선 제시한다. CA 신규 생성은 백업이 없는 경우에 한한다(5절 참조). **CA 백업 생성·보관은 설치 완료 조건**으로 운영 문서에 명시한다.
- [ ] CA 백업 파일은 암호화하며, 운영 데이터 백업과 동일한 접근 통제를 적용한다.

### 3.2 서버 서비스 (등록 시 발급)

- [ ] 서버 서비스는 **자체 키쌍을 로컬에서 생성**하고, Directory 등록 시 CSR을 제출하여 인증서를 발급받는다. **개인키는 어떤 경우에도 서버 밖으로 전송하지 않는다.** (Directory가 키쌍을 만들어 내려보내는 구조 금지)
- [ ] 발급 인증서의 **SAN에는 Directory가 식별한 서비스 제공 가능 NIC의 유효한 접속 주소 전체와 설정된 호스트명을 포함**한다. Loopback, Link-local, APIPA, 비활성·임시·컨테이너/가상 전용 인터페이스는 제외하며, 포함·제외 기준은 제품 정책으로 정의한다. Directory는 등록 정보를 기반으로 자동 기입한다.
  - *정책 결정: 클라이언트가 어느 서비스 경로로 접속할지 사전에 한정할 수 없으므로, 서비스 NIC에 한해서는 전체 포함으로 접속 실패 방지를 우선한다.*
- [ ] 서버 인증서 유효기간은 **1~3년**으로 하고, **만료 전 자동 재발급**을 구현한다(등록 채널을 통한 무인 갱신). 내부 CRL(4.4)을 사용할 수 없는 구성에서는 1년 이하를 우선 검토한다.
- [ ] 서버 주소 변경(NIC 추가·IP 변경)을 감지하면 자동으로 재발급을 요청한다. (SAN 불일치로 인한 접속 실패 방지)
- [ ] Directory는 발급 인증서의 **Serial을 발급 대장으로 관리하고 중복을 방지**한다. 발급·재발급·거부·폐기 이벤트는 감사 로그로 남긴다.
- [ ] 발급 인증서 프로파일: 서버 인증서는 **EKU에 Server Authentication**, Key Usage는 **digitalSignature를 기본**으로 한다(keyEncipherment는 TLS 1.2 RSA 키 교환을 의도적으로 지원하는 경우에만, keyAgreement는 정적 DH/ECDH를 실제 사용하는 경우에만 추가). CA 인증서는 **Basic Constraints CA=TRUE(경로 길이 0)**, Key Usage keyCertSign·cRLSign으로 발급한다. 서명 알고리즘은 SHA-256 이상, 키 강도는 RSA 2048 이상 또는 ECDSA P-256 이상으로 한다.

## 4. 클라이언트 — 검증 및 핀 배포

### 4.1 검증 조건

- [ ] 클라이언트의 신뢰 앵커는 **Directory CA 하나**다. 클라이언트는 **CA 인증서(공개키 포함)와 그 SPKI SHA-256 핀을 함께 저장**한다. (핀 해시만으로는 서명 검증이 불가능하므로 CA 인증서 자체를 보관하며, 서버가 TLS 체인에 CA 인증서를 포함해 보낸다고 가정하지 않는다) 서버별 핀은 사용하지 않는다.
- [ ] 서버 서비스 접속 시 다음을 **모두** 검증한다:
  1. **저장된 CA 공개키로** 서버 인증서(체인 각 단계)의 서명 검증
  2. 체인 상 CA의 SPKI가 **저장된 핀과 일치**하는지
  3. **SAN이 접속 주소와 일치**하는지
  4. 서버 인증서·CA 인증서의 **유효기간**
  5. 서버 인증서 **EKU에 Server Authentication** 포함 여부 및 Key Usage 적합성
  6. CA 인증서의 **Basic Constraints CA=TRUE** 여부
  7. 서명 알고리즘·키 강도가 3.2의 최소 기준 이상인지
  8. 서버 인증서가 **내부 CRL(4.4)에 폐기 등재되지 않았는지**
- [ ] 핀 검증은 인증서 전체 해시가 아니라 **공개키(SPKI) 해시**로 수행한다. (인증서 재발급 시에도 키가 유지되면 핀이 유효)

### 4.2 핀 최초 배포 (부트스트랩)

- [ ] 다음 순서로 자동화한다:
  1. **접속정보 파일(우선)** — Directory가 설치 시 **Directory 접속 주소 + 사이트 식별자 + CA 인증서 + CA SPKI 핀**이 담긴 접속정보 파일을 자동 생성하고, 클라이언트 설치 패키지 구성 또는 최초 설정 시 이 파일을 읽어 신뢰 정보를 등록한다. 파일은 설치 패키지 포함 또는 접근 제한된 공유 폴더로 전달한다.
  2. **TOFU(폴백)** — 접속정보 파일 없이 최초 접속하는 경우, 수신한 **CA 인증서와 SPKI 핀을 함께 자동 신뢰·저장**한다. 이후 접속부터는 저장된 핀과 불일치 시 5절의 재신뢰 정책을 따른다.

### 4.3 TOFU 잔여 위험 (수용)

- 최초 접속 시점에 중간자가 개입해 있으면 공격자의 CA를 신뢰하게 되는 위험이 있다. **폐쇄망 구축 환경(구축 시점에 공격자가 경로에 존재할 가능성이 극히 낮음)을 전제로 이 위험을 수용**하며, 각 프로젝트는 적용표에 잔여 위험 수용으로 기록한다.
- 완화 수단: 접속정보 파일 방식이 가능한 현장에서는 항상 파일 방식을 우선한다. TOFU로 등록된 핀은 관리 화면에서 지문을 조회할 수 있게 하여 사후 대조가 가능하도록 한다.

### 4.4 인증서 폐기 (내부 CRL)

- [ ] Directory는 **표준 X.509 CRL을 자체 CA로 서명하여 내부망에서 제공**한다. 자체 포맷의 폐기 목록을 새로 정의하지 않는다. (표준 CRL은 발급자 식별·서명·thisUpdate/nextUpdate·CRL Number를 이미 포함하며, 클라이언트 라이브러리의 검증 지원을 그대로 활용한다) 관리 기능에서 특정 서버 인증서를 폐기(발급 대장 기준)할 수 있어야 한다.
- [ ] 클라이언트는 CRL의 **서명을 저장된 CA 공개키로 검증한 후 캐시**하고, **nextUpdate를 캐시 유효기간으로 사용**한다. 서명이 검증되지 않거나 보유한 것보다 CRL Number가 낮은(오래된) CRL은 수락하지 않는다.
- [ ] 유효한 CRL(캐시 포함)을 확보하지 못한 상태의 동작은 다음 2단계로 한다:
  1. **과거 검증에 성공한 이력이 있는 동일 인증서의 재접속은 허용하고 이벤트 로그를 남긴다.** (관제 가용성 우선 — 잔여 위험으로 기록)
  2. **처음 보는 인증서 또는 변경된 인증서는 차단한다.**
- [ ] 폐기된 서버 인증서의 서버는 재등록·재발급 전까지 정상 클라이언트의 신뢰를 받을 수 없다. 폐기 이벤트는 감사 로그 대상이다.

### 4.5 핀 저장 및 계획 교체

- [ ] 저장된 핀은 일반 사용자가 수정할 수 없는 위치·권한으로 보호한다. (공통 가이드의 폴더 ACL·로컬 시크릿 기준 준용)
- [ ] **계획된 CA 교체(만료 임박·키 교체)는 핀 롤오버로 무중단 처리한다**: Directory가 차기 CA 인증서를 현행 CA 서명이 검증되는 채널(기존 TLS 세션)로 클라이언트에 사전 전달하고, 클라이언트는 현행 핀 + 차기 핀을 함께 보유한 상태로 전환기를 거친다. 검증되지 않은 경로로 수신한 핀은 저장하지 않는다. 전환 기간에 접속하지 않아 롤오버를 놓친 클라이언트는 5절의 복구 절차(접속정보 파일 재배포 또는 재설치)를 따른다.

## 5. CA 변경 대응 (재설치·백업 유실·핀 불일치)

> 계획된 교체는 4.5의 핀 롤오버로 처리한다. 이 절은 **비계획 변경**(Directory 재설치·CA 백업 유실)과 **롤오버를 놓친 클라이언트**(전환 기간 장기 미접속 등)의 복구를 다룬다.

- [ ] 클라이언트는 핀 불일치 감지 시 **접속을 차단**한다. 핀 불일치를 새 핀으로 자동 교체하는 동작은 금지한다. (자동 재신뢰는 "정당한 재설치"와 "중간자 공격"을 구분하지 못하므로 핀 검증 자체를 무효화한다)
- [ ] 차단 화면에는 원인("서버 재구성으로 인한 인증서 변경 가능성")과 **복구 절차를 안내**한다. 사용자에게 지문 확인·판단을 요구하지 않는다.
- [ ] 복구 수단은 다음 두 가지로 한다:
  1. **접속정보 파일 재배포** — Directory가 새로 생성한 접속정보 파일을 클라이언트가 읽어 신뢰 정보를 교체한다. (재설치 불필요, 파일 배포가 가능한 현장의 우선 수단)
  2. **클라이언트 재설치(또는 신뢰 초기화)** — 클라이언트를 제거 후 재설치하면 저장된 신뢰 정보가 **완전히 삭제**되고, 재설치 후 4.2의 부트스트랩(접속정보 파일 → TOFU)이 처음부터 다시 수행되어 새 CA를 받는다. 재설치와 동등하게 동작하는 "신뢰 초기화" 기능을 클라이언트에 두는 것을 허용한다(관리자 권한 필요).
- [ ] 위 요건을 위해 **제거(uninstall) 및 신뢰 초기화 시 저장된 CA 인증서·핀·CRL 캐시를 반드시 완전 삭제**한다. 신뢰 정보가 잔존하면 재설치 후에도 핀 불일치 차단이 반복되므로, 제거 동작의 신뢰 정보 삭제를 릴리스 검증 항목에 포함한다.
- [ ] 재설치·신뢰 초기화를 통한 재신뢰는 TOFU 잔여 위험(4.3)과 동일한 위험 수용에 해당하며, 프로젝트 적용표에 기록한다.
- [ ] 서버 서비스는 Directory의 CA 변경을 감지하면(발급 CA 불일치) **재등록 절차를 통해 새 CA로 인증서를 재발급**받는다. 재등록에는 최초 등록과 동일한 인증(등록 토큰 등)을 요구한다.
- [ ] 이 절의 복구 절차는 백업을 생략할 근거가 아니다. CA 백업 복원(3.1)이 항상 표준 절차다.

## 6. 고객 CA 대응 (선택 기능)

고객사가 자체 PKI(AD CS 등)를 요구하는 현장을 위한 옵션이며, 두 모드는 독립적이다. 기본값은 Directory CA + 핀 검증이다.

### 6.1 고객 하위 CA 모드

- [ ] Directory가 CA 개인키를 **로컬에서 생성**하고, CSR만 고객 CA(AD CS 등)에 전달한다.
- [ ] 고객 CA는 **CA=TRUE(경로 길이 제한)인 하위 CA 인증서**를 발급하여 반환하고, Directory는 이를 자신의 CA 인증서로 사용한다. **고객 CA의 개인키는 Directory로 전달되지 않는다.**
- [ ] 발급·핀 배포·검증 동작은 3~5절과 동일하다. (클라이언트는 Directory 하위 CA를 핀)

### 6.2 외부 서버 인증서 모드

- [ ] 고객이 발급한 **서버별 인증서(pfx)를 각 서버에 직접 지정**하며, Directory는 인증서 발급 기능을 수행하지 않는다.
- [ ] 클라이언트는 핀 검증 대신 **Windows 시스템 신뢰 저장소 기반 체인 검증 모드**로 전환한다. (고객 루트 CA가 도메인 정책 등으로 신뢰 저장소에 배포되어 있음을 전제)
- [ ] 이 모드에서 인증서 갱신·폐기 관리는 고객 PKI 정책을 따르며, 자사 제품의 관리 범위 밖임을 계약·운영 문서에 명시한다.
