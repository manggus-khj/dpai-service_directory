# 서비스 디렉토리 개발계획

> 설계 상태: 아키텍처·도메인 규칙 정리, 보안·일부 API 계약 검토 필요
> 구현 상태: 미구현
> 최초 작성일: 2026-07-17
> 최종 정리일: 2026-07-17

Milestone XProtect용 자사 소프트웨어의 접속 경로를 관리하는 서비스 디렉토리의 제품 범위와 구현 계획을 정의한다. 요청·응답의 상세 계약은 [API 명세 안내](./서비스디렉토리_API명세.md)와 분리된 내·외부 명세를 따른다.

## 1. 목표와 범위

- Milestone XProtect 2021 R1 이상 Management Server와 함께 설치되는 x64 Windows 제품
- TCP `21000` 포트에서 서비스 디렉토리 API 제공
- 별도 DB 없이 XML 파일로 상태 영속화
- Windows Service로 동작하고 설치 파일 형태로 배포
- 등록·수정은 다른 애플리케이션의 요청과 운영자 승인으로만 반영
- 서비스 디렉토리 UI에서 승인 요청 없이 새 서비스 정보를 직접 만드는 기능은 제공하지 않음

핵심 기능:

1. 다른 애플리케이션의 신규 등록·수정 요청 접수
2. 운영자의 승인·거절·삭제
3. 제품코드별 승인 완료 서비스 정보 제공
4. 두 서비스 디렉토리 인스턴스 사이의 데이터 동기화
5. 서비스 상태 감시와 제한된 로컬 서비스 제어

현재 저장소에는 문서만 있다. 이 계획의 기술 선택은 설계 결정이며 빌드, 설치 또는 Milestone 환경에서 검증 완료된 상태가 아니다.

## 2. 결정 상태

### 2.1 확정된 제품 결정

| 항목 | 결정 | 근거·주의 |
|---|---|---|
| 제품·빌드 버전 | 제품 `1.0.0`, 초기 build `1` | 제품 버전은 사용자 명시 요청 때만 변경. 이후 새 변경의 commit+push 전달마다 build를 1 증가하며 같은 전달의 amend·push 재시도에는 중복 증가 금지. 단일 원본은 루트 `VERSION` |
| 언어·런타임 | C# + .NET Framework 4.8, x64 전용 | 대상 Windows 제품군과의 일관성. x86·AnyCPU 산출물은 지원하지 않으며 런타임 제공 전제는 실제 환경 검증 필요 |
| 지원 Milestone | XProtect 2021 R1 이상 | 2021 R1을 지원 하한으로 고정하고 이후 버전은 실제 설치·회귀 검증 대상으로 관리 |
| Windows 운영 환경 | AD 도메인과 Workgroup 모두 | 도메인 전용 계정·Kerberos·그룹 정책을 필수 전제로 두지 않음 |
| 메인 서비스 | Windows Service(`ServiceBase`) + `HttpListener` | 별도 웹 프레임워크 없이 Windows 서비스로 호스팅. 경로별 인증 계약은 내·외부 API 명세를 따름 |
| 트레이 UI | WPF + H.NotifyIcon | 사내 WPF 기준과 Windows 트레이 지원 |
| 와치독 | 별도 경량 Windows Service | 메인 프로세스·리스너 장애를 외부에서 감시 |
| 저장소 | XML + `XmlSerializer` | DB 없이 배포. 원자 교체뿐 아니라 다중 파일 복구 설계 필요 |
| 설치 | Inno Setup | 서비스·방화벽·자동 시작을 하나의 설치 흐름으로 관리 |
| API 포트 | TCP `21000` | 제품 간 고정 접속 계약 |
| 전송 프로토콜 | 폐쇄망 HTTP/1.1, TLS/HTTPS 미사용 | 인증서 발급·배포·갱신이 어려운 운영 환경에 대한 §8의 승인된 예외 |
| ProductCode | `[A-Z0-9]{4}` 형식의 4바이트 ASCII | trim·대문자 정규화 후 `OrdinalIgnoreCase` 유일 키로 사용 |
| External 일일 API 키 | ProductCode+시스템 로컬 `yyyyMMdd`를 날짜 파생 AES key로 암호화한 44자 Base64 | 별도 secret 배포 없이 매 요청 새 IV로 생성. 상세 계약은 외부 API 명세 |
| Admin 인증·인가 | `127.0.0.1` loopback `HttpListener`의 Negotiate와 로컬 `DEEPAi-ServiceDirectory-Operators` 그룹 | IP literal에서는 Kerberos를 전제로 하지 않고 AD·Workgroup 모두 NTLM을 허용. 검증된 hostname·SPN 환경만 Kerberos 선택 가능, 연결 단위 NTLM 자격 증명 재사용 금지 |
| Peer 페어링·인증 | ECDH P-256 일회성 키 교환, 양쪽 8자리 SAS 확인, DPAPI 저장, HMAC-SHA256 요청·응답 서명 | AD·Workgroup 공통 계약. External 일일 API 키를 재사용하지 않음 |
| 네트워크 노출 | 고정 원격 IP·CIDR allowlist 미사용 | 승인된 폐쇄망 인터페이스에만 바인딩하고 방화벽은 Domain·Private 프로필만 허용하며 Public 프로필은 차단 |
| 로컬 listener 주소 | 설치 시 선택한 단일 IP literal `ListenAddress` | interactive 설치는 활성 Domain·Private 인터페이스 주소 중 하나를 요구하고 unattended 설치는 명시적 인수를 요구. 자동 선택·wildcard fallback 금지 |
| 지원 규모 | 활성 서비스 1,000개, 외부 호출 저빈도 | 외부 앱은 통상 최초 로그인 시 1회 조회. 톰스톤은 활성 수에 포함하지 않고 sync에서 batch 처리 |
| 주기 동기화 | 10분 | 즉시 동기화 실패 보정 |
| 즉시 동기화 | 등록 승인, 수정 승인, 삭제 직후 | 정상 시 두 인스턴스의 지연 최소화 |
| 시계 편차 한계 | 60초 초과 시 거부 | 벽시계 기반 병합의 최소 안전 조건. 60초 이내 위험은 별도 검토 |
| 트레이 시작 | 사용자 로그인 시 자동 실행 | 상태·승인 UI 접근성 |
| 톰스톤 정리 | 시간으로 제거하지 않음. 같은 제품코드의 신규 등록 승인 때만 대체 | 장기 단절 뒤 삭제 항목 부활 방지 |

WinForms, 서비스 내장 웹 UI, Electron/Tauri, Avalonia/MAUI는 현재 범위에서 제외한다. 기술 선택을 바꾸거나 새 런타임·UI 스택을 추가하려면 근거와 영향 범위를 먼저 문서화한다.

### 2.2 외부 환경 전제

Milestone XProtect 2021 R1 이상과 x64는 지원 범위로 확정했다. 다만 지원할 정확한 Windows 버전 목록과 각 Milestone·Windows 조합에서 .NET Framework 4.8이 제공되는지는 저장소에서 검증할 수 없는 배포 환경 전제이므로 실제 환경에서 확인한 뒤 설치 요구사항으로 고정한다.

### 2.3 구현 차단 결정

| 영역 | 결정이 필요한 내용 |
|---|---|
| 지원 환경 | 지원 Windows 버전 목록과 Milestone 2021 R1 이상 조합별 .NET Framework 4.8·x64 설치 검증 |
| 프로젝트 기반 | 솔루션·프로젝트 구조, 의존성 버전과 재현 가능한 build/test/package 진입점 |
| 저장 복구 | XML `SchemaVersion`, 마이그레이션과 여러 파일 변경용 트랜잭션 저널 형식 |
| 외부 등록 결과 | 승인·거절 상태 조회 방식과 처리 이력 보존 기간 |
| 계약 버전 | 내·외부 API 버전 경로·협상, HTTP 상태 최종 매핑과 XML namespace 정책 |
| External 키 버전 | 일일 API 키 알고리즘 변경 시 신·구 버전 병행·폐기 정책 |
| 병합 버전 | 벽시계 LWW 위험 수용 또는 논리 버전 도입 |
| 운영 | 로그 보존 기본값·최대값, 별도 보안 감사 sink, 제거 시 데이터 기본 처리, 오프라인 패치와 디버그 심볼 정책 |

이 표의 항목을 임시 규칙이나 과도한 서비스 권한으로 대체하지 않는다. 평문 HTTP는 [§8.2](#82-tls-미사용-예외-기록), External 일일 API 키는 [§8.3](#83-external-일일-api-키-예외-기록)의 승인된 범위에서만 사용한다. Admin과 Peer는 [내부 API 명세](./서비스디렉토리_내부_API명세.md#2-신뢰-경계와-인증-계약)의 확정된 인증 계약을 따른다.

## 3. 구성요소와 책임

### 3.1 메인 서비스

- 승인된 디렉토리, 톰스톤, 승인 대기 요청과 동기화 설정의 유일한 소유자
- 외부, Admin, Peer API 처리
- 기동 시 검증된 XML을 메모리에 적재하고 조회는 메모리에서 처리
- 변경 시 원자적·복구 가능한 방식으로 영속화
- 동기화 스케줄과 병합 수행
- UI 없음

### 3.2 와치독

- `ServiceController`로 메인 서비스 상태 확인
- `GET /api/health`로 리스너 생존 여부 확인
- 정책상 허용된 실패에서 메인 서비스 재시작과 이벤트 기록
- Named Pipe로 제한된 서비스 제어 명령 수신
- 디렉토리 데이터와 승인·동기화 설정은 변경하지 않음
- 메인 서비스와 상호 의존하지 않고 먼저 시작해 나중에 종료

health는 10초 주기로 호출하고 요청 timeout은 3초다. 프로세스 상태와 health 실패가 3회 연속일 때만 재시작을 판단하며, 10분 안에 3회 재시작한 뒤에는 자동 재시작을 중단하고 운영자 경고 상태로 전환한다.

와치독을 `LocalSystem`으로 실행한다고 미리 고정하지 않는다. 필요한 서비스 제어 권한만 가진 계정을 우선 설계하고, 불가능할 때만 승인된 보안 예외를 기록한다.

### 3.3 서비스 컨트롤 트레이 앱

- 일반 사용자 권한으로 로그인 시 자동 실행
- 트레이 메뉴: 시작, 종료, 재시작, 동기화, 상태 모니터
- 상태 모니터:
  - 승인된 등록 목록과 확인 후 삭제
  - New·Modify 승인 대기 목록과 요청값·현재값 차이
  - 승인·거절
  - 피어 설정, 동기화·해제, 마지막 결과와 시계 편차
  - 설정: 시스템 로그 보존기간(일) 조회·변경
  - 메인 서비스와 와치독 연결 상태
- 상태 창이 열려 있을 때 sync 상태는 5초, 서비스·대기 목록은 10초 주기로 갱신하고 창이 숨겨지면 목록 polling을 중단
- XML 파일과 피어 API에 직접 접근하지 않음

권한이 필요한 작업은 와치독 또는 인증된 Admin API에 위임한다. UAC를 피하려고 IPC 인가를 생략하지 않는다.

### 3.4 설치 프로그램

- 실행 파일을 `%ProgramFiles%\DEEPAi\ServiceDirectory\` 아래에 설치
- 상태와 로그를 제한된 ACL의 `%ProgramData%\DEEPAi\ServiceDirectory\` 아래에 저장
- 두 서비스를 독립 자동 시작으로 등록
- TCP `21000` 인바운드는 Domain·Private 방화벽 프로필에서만 허용하고 Public 프로필에서는 차단. 고정 원격 IP·CIDR allowlist는 구성하지 않음
- interactive 설치에서는 활성 Domain·Private 네트워크 인터페이스의 non-loopback IPv4·IPv6 literal 하나를 `ListenAddress`로 선택하게 하고, unattended 설치에서는 명시적 `/ListenAddress=` 값을 필수로 받음. 유효한 주소가 없으면 설치를 중단하고 wildcard로 대체하지 않음
- 선택 주소의 exact URL ACL을 메인 서비스 SID에 부여하고 loopback URL ACL, 프로그램·포트 방화벽 규칙과 `config.xml`을 함께 구성. 주소 변경은 installer repair에서 서비스를 멈춘 뒤 검증·변경·재기동하며 실패 시 이전 설정으로 롤백
- 로컬 운영자 그룹 `DEEPAi-ServiceDirectory-Operators`를 만들고 AD 사용자·그룹 또는 Workgroup 로컬 사용자를 구성 가능하게 함
- 메인과 와치독은 각각의 Windows 가상 서비스 계정과 서비스 SID로 실행하고 필요한 파일·서비스 제어 ACL만 부여
- 트레이 로그인 자동 시작 등록
- 제거 시 서비스와 방화벽 규칙 제거, 데이터 보존 여부를 명시적으로 처리

경로의 회사명은 `DEEPAi`로 확정한다. 제품 디렉터리 이름은 `ServiceDirectory`를 사용한다.

## 4. 통신 아키텍처

```text
[트레이 앱: 일반 권한]
   │  Named Pipe: 서비스 제어
   ▼
[와치독: 최소 권한 서비스] ── 상태 확인·헬스체크 ──▶ [메인 서비스: TCP 21000]
                                                        ▲          │
[트레이 앱] ── loopback Admin API ──────────────────────┘          │ Peer API
                                                                   ▼
                                                     [상대 메인 서비스]

[다른 애플리케이션] ── External API ───────────────────▶ [메인 서비스]
```

| 경로 | 계약의 단일 원본 | 보안 경계 |
|---|---|---|
| 다른 앱 → 메인 | [외부 애플리케이션 API](./서비스디렉토리_외부애플리케이션_API명세.md) | 폐쇄망 HTTP + 44자 일일 API 키 + ProductCode 일치 + Domain·Private 방화벽 프로필 |
| 트레이 → 메인 | [내부 API의 Admin](./서비스디렉토리_내부_API명세.md#4-admin-api) | loopback HTTP + Negotiate Windows identity + 로컬 운영자 그룹 인가 |
| 메인 ↔ 피어 | [내부 API의 동기화](./서비스디렉토리_내부_API명세.md#5-피어-동기화-데이터) | ECDH+SAS 페어링, DPAPI pair root, HMAC-SHA256 서명·freshness·재전송 방지 + 설정된 피어 검증 |
| 트레이 → 와치독 | [내부 API의 Named Pipe](./서비스디렉토리_내부_API명세.md#6-named-pipe-서비스-제어-계약) | 로컬 운영자 그룹 ACL + 명령 허용 목록 |

## 5. 데이터와 저장

### 5.1 도메인 레코드

| 필드 | 소유권과 의미 |
|---|---|
| `Name` | 애플리케이션이 요청하고 운영자가 승인한 표시 이름. trim 후 1~128 Unicode scalar, UTF-8 최대 512바이트, 제어문자 금지 |
| `ProductCode` | trim 후 `ToUpperInvariant()`로 정규화한 `[A-Z0-9]{4}` 형식의 정확히 4바이트 ASCII. `OrdinalIgnoreCase`로 비교하는 유일 키 |
| `ServerAddress` | 승인된 IPv4·IPv6 literal 또는 최대 253자 ASCII DNS hostname. scheme·path·query·port 포함 금지 |
| `Port` | `1..65535` 포트 |
| `LastModifiedUtc` | 활성 레코드의 병합 비교 시각 |
| `Deleted`, `DeletedUtc` | 삭제 전파를 위한 톰스톤과 비교 시각 |
| `OriginInstanceId` | 마지막 변경 출처. 동시각 병합의 결정적 타이브레이크 |

`OriginInstanceId`는 기존 초안에서 빠져 있던 내부 메타데이터다. 현재 송신자 ID만으로 타이브레이크하면 전달 경로에 따라 결과가 달라질 수 있어 레코드에 보존한다. 외부 조회 DTO에는 노출하지 않는다.

### 5.2 파일

| 파일 | 내용 |
|---|---|
| `directory.xml` | 승인된 활성 레코드와 톰스톤 |
| `pending.xml` | New·Modify 승인 대기 요청 |
| `config.xml` | 단일 IP literal `ListenAddress`, 동기화 상태, 피어 정보, 설치 `InstanceId`, 마지막 발급 peer epoch `LastPeerKeyEpoch`, 로그 보존기간 `LogRetentionDays` |
| `secrets/peer.dat` | ECDH 페어링으로 파생한 pair root와 peer identity·key epoch의 DPAPI LocalMachine 보호값 |

모든 XML 루트에는 `SchemaVersion`을 두며 구체 버전과 마이그레이션 표는 구현 전에 확정한다. 외부 등록 결과 이력 저장 방식도 함께 추가한다. `secrets/peer.dat`은 메인 서비스 SID, `SYSTEM`, Administrators만 읽을 수 있고 다른 머신 복원은 지원하지 않는다.

세 파일은 `%ProgramData%\DEEPAi\ServiceDirectory\` 데이터 루트 아래에 둔다. 로그 파일은 §9의 하위 경로를 사용한다.

### 5.3 저장 불변식

- 기동 시 한 번 로드하고 스키마·필수 값·중복 키를 검증한 뒤 메모리에 게시한다.
- 변경이 있을 때만 같은 볼륨의 임시 파일에 작성하고 flush한 뒤 원자적으로 교체하며 백업을 유지한다.
- 최초 파일 생성은 대상이 있어야 하는 `File.Replace`와 별도 경로로 처리한다.
- 파일 핸들을 상시 유지하지 않는다.
- 디렉토리·대기·설정의 모든 변경 명령과 sync 최종 병합·게시는 인스턴스당 하나의 state mutation gate로 직렬화한다. 네트워크 송수신과 XML parse는 gate 밖에서 수행하고, gate 안에서 현재 revision을 다시 검증한 뒤 저널·원자 저장과 immutable snapshot 교체를 완료한다.
- 조회는 현재 immutable snapshot을 사용해 mutation gate를 점유하지 않는다. sync 네트워크 교환 중 생긴 로컬 변경은 최종 병합 시 현재 snapshot에 보존하고 다음 sync 사이클에서 피어로 전파한다.
- 승인처럼 `directory.xml` 갱신과 `pending.xml` 제거가 함께 필요한 작업은 파일별 원자 교체만으로 충분하지 않다. 중간 종료 뒤 중복·유실 없이 복구하는 저널 또는 동등한 crash-consistency 설계를 먼저 확정한다.
- 손상 파일, 백업 복구, 디스크 부족, 권한 실패와 강제 종료를 명시적으로 처리하고 오류를 숨기지 않는다.
- 트레이, 와치독, 다른 애플리케이션은 파일을 직접 읽거나 수정하지 않는다.
- 정상 지원 규모는 활성 서비스 1,000개다. `Deleted=true` 톰스톤은 활성 수에는 포함하지 않으며 임의 삭제하지 않고 안정된 sync snapshot을 batch로 나눈다.
- 승인 대기는 ProductCode당 하나, 인스턴스당 전체 1,000개로 제한하고 800개부터 운영 경고를 표시한다.

## 6. 등록·수정·삭제 불변식

외부 요청 처리의 상세 계약은 [외부 API 명세](./서비스디렉토리_외부애플리케이션_API명세.md#43-post-apiregistration)에 둔다.

아래 표는 위에서 아래 순서로 처음 일치하는 행 하나만 적용한다.

| 우선순위 | 현재 상태 | 요청 | 도메인 결과 |
|---|---|---|---|
| 1 | 같은 ProductCode의 대기 요청 존재 | 대기 요청과 동일 | 기존 대기 요청 재사용 |
| 1 | 같은 ProductCode의 대기 요청 존재 | 대기 요청과 다름 | 충돌, 현재 활성값과 같아도 자동 교체 금지 |
| 2 | 대기 없음, 미등록 또는 톰스톤 | 유효한 요청 | New 승인 대기 |
| 2 | 대기 없음, 활성 | 현재 값과 동일 | 변경 없음 |
| 2 | 대기 없음, 활성 | 현재 값과 다름 | Modify 승인 대기 |

- 등록·수정은 승인 전에는 정식 디렉토리에 반영하지 않는다.
- 승인 대기는 서비스 재시작 뒤에도 유지한다.
- 대기 생성 시 해당 키의 현재 상태를 나타내는 불변 base revision 또는 동등한 스냅샷을 함께 보존한다.
- 승인 대기는 피어와 동기화하지 않는다. 요청을 받은 인스턴스에서만 처리하고 승인 결과만 동기화한다.
- 승인 시 현재 revision이 base와 다르면 요청값이 이미 활성값과 같은 경우만 “이미 충족”으로 정리한다. 그 외에는 충돌로 남기고 자동 덮어쓰기·재분류하지 않는다.
- 거절은 대기 요청만 처리하며 디렉토리·톰스톤을 변경하지 않는다.
- 삭제는 물리 제거 대신 톰스톤을 만들고 즉시 동기화를 예약한다.
- 같은 제품코드의 신규 등록 승인만 기존 톰스톤을 새 활성 레코드로 대체할 수 있다.
- 외부 조회는 톰스톤을 없는 항목으로 취급한다.
- `Deleted=false`인 활성 서비스는 최대 1,000개다. 1,000개 상태에서는 기존 활성 항목의 Modify와 삭제는 허용하지만 새 ProductCode의 New 승인은 `LIMIT_EXCEEDED`로 거부하고 대기를 유지한다.

## 7. 동기화 불변식

실행 시점:

1. 등록·수정 승인 또는 삭제 직후
2. 즉시 동기화 실패 보정을 위한 10분 주기
3. 서비스 기동 시
4. 운영자 수동 요청

모든 사이클은 피어 인증과 handshake부터 시작한다. 데이터는 ProductCode별로 병합하며 활성 항목은 `LastModifiedUtc`, 톰스톤은 `DeletedUtc`를 비교한다. 동시각·상이한 값은 레코드의 `OriginInstanceId`로 결정적으로 판정한다. 비교 시각과 출처가 같은데 payload가 다르면 손상된 revision으로 보고 전체 병합을 중단한다.

다음 조건을 지킨다.

- sync 송수신은 일관된 불변 스냅샷을 사용하고 피어별 활성 session을 하나로 제한한다. 최종 병합·게시만 공통 state mutation gate에서 현재 로컬 snapshot과 수행해 관리 변경을 잃지 않는다.
- 한 sync batch는 최대 1,000개 레코드이며 톰스톤 때문에 초과하면 같은 `SnapshotId`의 여러 batch로 교환한다. 전체 batch 검증·병합이 끝나기 전에는 부분 상태를 게시하지 않는다.
- 한쪽에만 있는 톰스톤도 전파한다.
- 원격 스냅샷을 병합한 후보의 활성 서비스가 1,000개를 넘으면 전체 exchange를 `DIRECTORY_CAPACITY`로 거부하고 현재 상태를 게시·저장하지 않는다. 운영자가 한쪽에서 충분한 활성 서비스를 삭제한 뒤 다시 sync한다.
- 해제 통지 실패와 로컬 해제 성공을 구분해 기록한다.
- 로컬 변경 시각은 시계 역행에도 이전 로컬 변경보다 단조 증가시킨다.
- 시계 편차가 60초를 넘으면 병합하지 않고 상태 모니터에 오류를 표시한다.
- 60초 이내 편차에서도 실제 사건 순서와 LWW 결과가 달라질 위험은 별도 결정이 필요하다.

최초 페어링은 양쪽 로컬 운영자가 5분 페어링 창을 열고 ECDH P-256 공개키를 교환한 뒤, 각 트레이에 독립적으로 계산된 8자리 SAS가 같은지 양쪽에서 확인하는 상태 머신을 사용한다. 양쪽 확인과 MAC된 commit이 끝나면 `PairedDisabled`가 되고, 운영자가 sync를 활성화해야 `Enabled`가 된다. 상세 계약은 [내부 API 명세](./서비스디렉토리_내부_API명세.md#53-최초-페어링과-key-epoch)를 따른다.

### 8.2 TLS 미사용 예외 기록

| 항목 | 결정 |
|---|---|
| 판정 | `예외 승인` |
| 승인 근거 | 사용자의 명시적 TLS 미사용 결정(2026-07-17) |
| 승인자 기록 | 릴리스 전 프로젝트 예외 승인 기록에 승인자의 성명·역할을 기입해야 함 |
| 범위 | TCP `21000`의 External·Health·Peer API와 loopback Admin API. 기준 scheme은 `http`이며 HTTPS listener, 인증서 바인딩과 HSTS를 구성하지 않음 |
| 사유 | 배포 대상이 외부·비신뢰 네트워크와 라우팅되지 않는 폐쇄망이며 인증서 발급·배포·갱신·폐기 체계를 안정적으로 운영하기 어려움 |
| 수용 위험 | 네트워크 또는 호스트 경계가 침해되면 트래픽 도청·변조, 엔드포인트 사칭, 자격 증명 탈취와 서비스 정보 노출 가능성이 TLS 사용 때보다 커짐 |
| 적용 기간 | 폐쇄망 전용 배포 조건이 유지되는 동안. 각 릴리스와 망 구성 변경 시 재검토 |

필수 보완 통제:

- TCP `21000`은 승인된 폐쇄망 인터페이스에만 바인딩하고 Windows 방화벽의 Domain·Private 프로필에서만 허용하며 Public 프로필에서는 차단한다. 고정 원격 IP·CIDR allowlist는 사용하지 않지만 External·Peer에 wildcard `http://+:21000/` 또는 `0.0.0.0` 바인딩을 사용하지 않는다.
- Admin API는 `127.0.0.1` loopback에만 바인딩한다. External·Peer API를 인터넷, 사내 일반망 또는 신뢰되지 않은 구간으로 포워딩하지 않는다.
- External·Health 요청은 §8.3의 일일 API 키 예외를 적용한다. Admin은 loopback 호출자 인증·운영자 인가, Peer는 상호 인증·메시지 무결성·freshness·재전송 방지를 별도로 적용한다.
- Admin은 Windows identity와 로컬 운영자 그룹 외의 애플리케이션 자격 증명을 두지 않는다. Peer pair root는 소스·평문 XML·Credential Manager로 분산하지 않고 내부 API 계약대로 `secrets/peer.dat`의 DPAPI `LocalMachine` 보호와 발급·교체·폐기 절차를 사용한다.
- 인증·인가 실패와 네트워크 정책 위반은 §9 시스템 파일과 분리된 보안 감사 대상으로 기록한다. 구체적인 sink는 구현 전에 확정한다.
- 설치·진단 단계에서 HTTPS나 인증서가 암묵적 전제가 되지 않는지, Public 방화벽 프로필과 비신뢰 인터페이스에서 접근이 차단되는지 검증한다.

이 보완 통제와 일일 API 키는 payload 기밀성을 제공하지 않는다. 평문 노출에 대한 잔여 위험은 승인 범위에 포함한다.

다음 중 하나가 발생하면 이 예외는 효력을 잃으며 배포 전에 TLS 또는 동등한 보안 채널을 다시 설계한다.

- 외부망·인터넷·사내 일반망·제3자 네트워크와 연결 또는 라우팅
- 서비스 정보보다 민감도가 높은 개인정보·인증정보·영상 메타데이터의 전송 추가
- 폐쇄망 내부의 비신뢰 단말 또는 다중 테넌트 접근 허용
- 운영 가능한 인증서 또는 별도 보안 채널 수명주기 확보

### 8.3 External 일일 API 키 예외 기록

| 항목 | 결정 |
|---|---|
| 판정 | `예외 승인` |
| 승인 근거 | 사용자의 명시적 최초 제안 방식 적용 결정(2026-07-17) |
| 승인자 기록 | 릴리스 전 프로젝트 예외 승인 기록에 승인자의 성명·역할을 기입해야 함 |
| 범위 | 다른 애플리케이션용 External 세 엔드포인트와 같은 계약으로 와치독이 재사용하는 `GET /api/health`. Admin·Peer에는 적용하지 않음 |
| 방식 | `[A-Z0-9]{4}` ProductCode와 시스템 로컬 `yyyyMMdd`를 결합하고 날짜의 SHA-256을 AES-256 key로 사용하는 CBC/PKCS#7 암호문. 무작위 IV와 결합해 44자 Base64 헤더로 전송 |
| 사유 | 제품별 대칭 secret, 공개키 또는 인증서를 등록 애플리케이션에 안전하게 배포할 수 있는 채널이 없음 |
| 수용 위험 | 알고리즘과 ProductCode를 아는 주체의 자체 생성, 동일 ProductCode 호출자 구분 불가, 당일 replay, method·path·body 및 응답 무결성 미제공, HTTP 노출 |
| 적용 기간 | 폐쇄망 전용 배포와 현재 External API 계약이 유지되는 동안. 각 릴리스와 신뢰 경계 변경 시 재검토 |

상세 wire contract와 테스트 벡터의 단일 원본은 [외부 애플리케이션 API §2.3~§2.5](./서비스디렉토리_외부애플리케이션_API명세.md#23-일일-api-키-생성-계약)다.

필수 보완 통제:

- 서비스 조회와 등록 요청은 키에서 복원한 ProductCode가 쿼리·XML의 정규화된 ProductCode와 일치해야 한다.
- 서버의 현재 시스템 로컬 날짜만 허용하고 이전·다음 날짜 유예를 두지 않는다. 자정 실패 시 호출자가 새 날짜 키로 한 번 재시도한다.
- 폐쇄망 인터페이스 바인딩, Domain·Private 방화벽 프로필, endpoint별 rate limit과 동시 실행 제한을 적용한다. 고정 원격 IP·CIDR allowlist는 사용하지 않는다.
- 등록·수정은 유효한 일일 키만으로 확정하지 않고 기존 운영자 승인 절차를 거친다.
- 누락·복호화·padding·날짜·ProductCode 실패는 모두 같은 `401 INVALID_API_KEY`로 응답하고 키 원문과 상세 실패 단계를 로그에 남기지 않는다.
- 알고리즘 문서는 사내 승인 범위에서만 배포하고 공개 문서나 오류 응답으로 노출하지 않는다.
- 고정 AES secret, 난독화된 공용 secret 또는 별도 master key를 코드·설정에 추가하지 않는다.
- 이 방식은 강한 호출자 인증, 요청 서명 또는 replay 방지로 표현하지 않는다.

### 8.4 적용 원칙

- `해당 없음`은 보안 기준을 충족했다는 뜻이 아니라 현재 기능이나 공격 표면이 없다는 뜻이다.
- `미결정` 항목은 임시 무인증, 공용 시크릿 또는 넓은 권한으로 대체해 완료 처리하지 않는다.
- TLS와 External 일일 API 키 예외는 각각 §8.2·§8.3의 범위에만 적용한다. Admin·Peer와 나머지 하드닝 항목을 완화하지 않는다.
- 기능, 배포망 또는 신뢰 경계가 바뀌면 이 표와 예외 기록을 같은 변경에서 재평가한다.

## 9. 시스템 로그 정책

### 9.1 경로와 일 단위 파일

- 로그 루트: `%ProgramData%\DEEPAi\ServiceDirectory\logs\system\`
- 데이터 루트 기준 상대 경로: `logs/system/dpai-sd_yyyy-MM-dd.log`
- 실제 예: `%ProgramData%\DEEPAi\ServiceDirectory\logs\system\dpai-sd_2026-07-17.log`
- 파일명의 날짜는 UTC가 아니라 로그를 쓰는 시점의 **현재 시스템 로컬 날짜**다.
- 로컬 날짜가 바뀌면 새 날짜 파일로 전환한다.
- 각 이벤트는 UTF-8 텍스트 한 줄로 append한다.

### 9.2 시각 규칙

- 로그 레코드 시각은 `DateTimeOffset.Now` 의미의 현재 시스템 로컬 시각과 UTC offset을 사용한다.
- 형식: `yyyy-MM-ddTHH:mm:ss.fffzzz`
- 예: `2026-07-17T18:42:31.123+09:00`
- `Z` 또는 UTC로 변환한 시각을 로그에 쓰지 않는다.
- 실행 중 시스템 timezone 또는 DST offset이 바뀌면 다음 레코드부터 변경된 로컬 시각과 offset을 사용한다. 기존 파일은 이름을 바꾸지 않는다.
- 로컬 날짜 규칙은 시스템 로그와 External 일일 API 키에만 적용한다. API payload·동기화·도메인 병합 시각은 기존대로 UTC를 사용한다.

기본 한 줄 형식:

```text
{LocalTimestampWithOffset} [{EventCode}] {Details}
```

### 9.3 파일에 기록할 이벤트

| EventCode | 기록 시점 | 최소 Details |
|---|---|---|
| `SERVICE_STARTED` | 메인 서비스 초기화와 API listener 준비가 성공한 직후 | 서비스 인스턴스 식별자 |
| `SERVICE_STOPPED` | 정상 종료 절차가 완료되어 프로세스를 끝내기 직전 | 종료 사유 |
| `REGISTERED_SERVICE_CREATED` | 신규 등록 승인이 영속화된 직후 | `ProductCode` |
| `REGISTERED_SERVICE_UPDATED` | 등록 서비스 수정 승인이 영속화된 직후 | `ProductCode` |
| `REGISTERED_SERVICE_DELETED` | 등록 서비스 톰스톤이 영속화된 직후 | `ProductCode` |
| `SYNC_INITIAL_STARTED` | 페어링·동기화 활성화 후 최초 전체 동기화가 시작될 때 | 피어, 실행 원인 |
| `SYNC_STARTED` | 최초 동기화가 아닌 동기화 사이클이 시작될 때 | 피어, 실행 원인 |
| `SYNC_STOPPED` | 동기화 설정이 로컬에서 비활성화될 때 | 피어, 중지 원인 |
| `SYNC_SUCCEEDED` | 최초 또는 일반 동기화 사이클이 성공하고 결과가 영속화된 뒤 | 피어, 실행 원인 |

추가 규칙:

- `SYNC_INITIAL_STARTED`와 `SYNC_STARTED`는 같은 사이클에 중복 기록하지 않는다.
- 최초 사이클도 성공하면 `SYNC_SUCCEEDED`를 기록한다.
- 등록 요청 접수, 승인 전 대기, 거절은 위 파일 로그 대상이 아니다.
- 이벤트는 해당 상태 변경이 성공적으로 영속화된 뒤 기록한다. `SERVICE_STOPPED`만 프로세스 종료 전 기록한다.
- 비밀번호, 토큰, 인증서, 전체 XML, 서버 주소와 스택 트레이스를 Details에 기록하지 않는다.
- 위 목록 밖의 보안 감사·진단 이벤트를 같은 파일에 임의로 추가하지 않는다. 별도 sink가 필요하면 하드닝 적용표와 이 절을 먼저 갱신한다.

### 9.4 보존기간 설정

- `config.xml`의 `LogRetentionDays`에 보존할 로컬 달력 일수를 정수로 저장한다.
- 값은 1 이상이어야 하며 0, 음수, 정수가 아닌 값과 overflow는 거부한다.
- 트레이 설정 화면은 현재 값을 조회하고 일 단위 정수로 변경할 수 있어야 한다.
- 설정 변경은 Admin API를 통해 메인 서비스가 검증·영속화한다. 트레이가 `config.xml`을 직접 수정하지 않는다.
- 보존 정리는 서비스 시작, 로컬 날짜 전환 후 첫 로그 기록, 설정값 변경 직후 수행한다.
- 오늘을 포함해 최근 `LogRetentionDays`개의 로컬 날짜 파일을 보존한다.
- 정리 대상은 정확한 로그 디렉터리 안에서 `dpai-sd_yyyy-MM-dd.log` 형식과 일치하는 파일로 제한한다. 다른 파일이나 하위 경로는 삭제하지 않는다.
- 설치 기본값과 허용할 최대 일수는 아직 확정하지 않는다.

## 10. 설치 계획

- 코드 서명 인증서가 없으므로 실행 파일, 라이브러리와 설치 파일의 코드 서명은 현재 범위에서 제외한다. 서명 단계나 인증서 시크릿을 빌드에 추가하지 않는다.
- 트레이와 사용자가 직접 실행하는 일반 실행 파일은 `asInvoker`로 실행한다. 권한 상승은 설치 프로그램의 실제 설치 작업에만 제한한다.
- 설치 루트 `%ProgramFiles%\DEEPAi\ServiceDirectory\`와 데이터 루트 `%ProgramData%\DEEPAi\ServiceDirectory\`를 생성한다. 일반 사용자는 실행 파일을 교체할 수 없어야 하며 로그를 포함한 운영 데이터의 수정·삭제 권한은 역할별 최소 범위로 제한한다.
- `config.xml`에는 External 일일 API 키·고정 AES secret·별도 master key를 저장하지 않는다. Peer pair root는 별도 `secrets/peer.dat`에 DPAPI LocalMachine으로 보호하고 서비스 SID ACL을 적용한다.
- 메인·와치독은 각각의 Windows 가상 서비스 계정으로 구성하고, 메인 서비스 SID에는 데이터·로그와 DPAPI blob 접근만, 와치독 서비스 SID에는 필요한 메인 서비스 제어 권한만 부여한다.
- HTTP listener는 §8.2에 정의한 인터페이스에만 바인딩한다. 방화벽은 Domain·Private 프로필의 TCP `21000`만 허용하고 Public 프로필에서는 차단한다. 고정 원격 IP·CIDR allowlist는 만들지 않는다.
- 설치·repair는 `ListenAddress`가 현재 로컬 인터페이스에 할당된 non-loopback IPv4·IPv6 literal이고 네트워크 프로필이 Domain·Private인지 검증한다. 메인 서비스도 기동 때 다시 검증하며 누락·미할당·wildcard·Public 주소면 기동을 실패시키고 자동 주소 선택이나 wildcard fallback을 하지 않는다.
- 로컬 `DEEPAi-ServiceDirectory-Operators` 그룹과 Admin·Named Pipe ACL을 만들고 AD 사용자·그룹 또는 Workgroup 로컬 사용자를 구성할 수 있게 한다.
- 트레이 자동 시작은 사용자 로그인 단위로 등록한다.
- DLL 검색 경로를 제한하고 쓰기 가능한 다운로드·임시 경로에서 실행한 설치 프로그램의 DLL 하이재킹 가능성을 검증한다.
- DEP/NX, ASLR과 CFG 적용 여부를 확인한다. 운영 산출물에는 인증 우회 플래그, 테스트 코드와 개발자 백도어를 포함하지 않으며 디버그 심볼의 분리·보관 정책을 정한다.
- 자동 업데이트는 제공하지 않는다. 패치는 승인된 수동·오프라인 경로로 반입하고 설치 전에 SHA-256 이상 체크섬을 별도 신뢰 채널의 manifest와 대조한다. 코드 서명 부재로 출처 인증이 약해지는 잔여 위험을 배포 절차에 기록한다.
- 제거 시 데이터 보존 선택의 기본값과 ACL 유지·제거를 정의한다.
- 설치, 업그레이드, 롤백, 제거를 지원 Windows와 Milestone XProtect 2021 R1 이상 조합의 x64 환경에서 검증한다.

| 5. 와치독·설치 | 최소 권한 서비스 제어, 감시 정책, Inno Setup 패키지 |
| 6. 릴리스 검증 | 보안, 장애, 설치·제거, 지원 환경과 이중화 종합 테스트 |

존재하지 않는 솔루션·프로젝트 구조 또는 빌드 명령을 완료로 표시하지 않는다. 실제 파일을 만든 뒤 이 문서와 저장소 루트 지침을 갱신한다.

빌드·컴파일 테스트·패키징은 매 수정이나 commit+push 때 자동 실행하지 않고 사용자가 명시적으로 빌드 체크를 요청한 경우에만 수행한다. 빌드 체크 요청과 build 번호 증가는 별개이며, build 번호는 빌드를 실제 실행했는지와 관계없이 새 변경의 commit+push 전달 단위에 따라 증가한다.
