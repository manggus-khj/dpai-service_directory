# 저장소 공통 작업 지침

```text
최초 작성일: 2026-07-19
최종 변경일: 2026-07-20
revision: 10
```

## 1. 문서 체계와 역할 분담

### 1.1 문서 체계

- `AGENTS.md`(이 파일)는 저장소 공통 지침의 유일한 원본이다. 저장소 전체에 적용되는 일반 지침만 기록하며, 개별 앱·배포 패키지의 요구사항, 설계, 일정, 구현 단계는 기록하지 않는다. 개별 작업의 이유로 이 파일을 수정하지 않는다.
- `CLAUDE.md`에는 세부 지침을 복제하지 않고 `AGENTS.md`를 참조하라는 내용만 둔다. 공통 지침을 변경할 때는 `AGENTS.md`만 수정하고 참조 연결을 유지한다.
- `README.md`에는 앱의 명칭과 개발 단위, 계획 문서들의 용도·위치·파일명을 지정한다.
- 이 파일 상단에는 3.2의 문서 헤더 규칙을 동일하게 적용한다.

### 1.2 역할 분담

- **Codex**: 개발과 코드 수정을 담당한다. 구현은 3.5의 할 일 목록을 기준으로 수행한다.
- **Claude**: 계획 작성과 코드 검증을 담당한다. 계획 작성에는 3.5의 할 일 추출·갱신이 포함된다.
- Claude는 별도 언급이 없으면 항상 계획 작성까지만 수행하며, 구현(코드 작성·수정)을 스스로 진행하지 않는다. 요청에 구현이 필요한 내용이 포함되어 있어도 명시적인 구현 지시가 없으면 계획 제안으로 작업을 마친다.
- 사용자의 명시적인 구현 지시는 그 요청 1회에만 적용되는 일회성 허용이다. 이전 대화나 세션에서 구현한 이력이 있어도, 다음 요청부터는 다시 계획 전용 원칙으로 복귀한다.

## 2. 저장소 구조

표준 최상위 구조는 다음과 같다. 아래 규칙 외의 위치에 같은 성격의 파일을 두지 않는다.

| 위치 | 용도 |
|---|---|
| `/docs/plan/` | 모든 계획 문서와 할 일 목록 (하위 폴더 없이 파일명으로 구분) |
| `/src/<배포패키지명>/` | 코드 소스 |
| `/common/` | 빌드·인스톨러 생성용 공용 패키지 |
| `/artifacts/<배포패키지명>/` | 빌드 과정에서 생성되는 파일 |
| `/installer/` | 최종 설치패키지 (하위 폴더 없음) |

### 2.1 위치 규칙

- **계획 문서와 할 일 목록**: 모두 `/docs/plan/` 바로 아래에 둔다. 패키지·앱별 하위 폴더를 만들지 않으며, 구분은 3.1의 파일명 규칙으로 한다. `/docs/plan/` 이외의 위치에 계획 루트를 만들지 않는다. 하위 폴더는 이미지 등 md가 아닌 파일을 보관할 때만 허용하며, 그 파일들을 사용하는 주인 md 파일과 같은 이름의 폴더를 만들어 넣는다.
- **코드 소스**: `/src/<배포패키지명>/` 아래에 두고, 여러 패키지의 소스를 하나의 폴더에 섞지 않는다.
- **빌드·인스톨러 생성용 패키지**(빌드 도구, 인스톨러 제작 도구, 공용 컴포넌트, 재배포 가능한 런타임 등): `/src/`가 아닌 `/common/`에 두고 모든 배포 패키지가 공용으로 사용한다. 같은 컴포넌트를 패키지마다 중복 보관하지 않으며, 배포 패키지가 하나뿐이어도 위치를 바꾸지 않는다. 위치가 일정해야 존재 여부를 파악할 수 있기 때문이다.
  - 단, 해당 패키지의 개발사가 권장하는 설치 위치가 있는 경우에는 `/common/` 대신 그 권장 경로에 설치한다.
  - 특정 배포 패키지 전용의 빌드·인스톨러 설정 파일이나 아이콘 등의 자산은 이 규칙의 대상이 아니며, 기존 위치 규칙을 따른다.
- **빌드 생성 파일**: `/artifacts/<배포패키지명>/` 아래에 두고, 여러 패키지의 생성물을 섞지 않는다. 빌드 중간 파일을 최종 설치패키지로 취급하지 않는다.
- **설치패키지**: `/installer/` 바로 아래에 생성한다.

### 2.2 배포 패키지가 하나뿐인 경우의 간소화

배포 패키지가 하나뿐인 앱 또는 웹은 다음과 같이 간소화한다.

- 코드 소스는 배포패키지명 폴더 없이 `/src/` 바로 아래에 둔다.
- 계획 문서와 할 일 목록은 파일명에서 앱 이름 접두를 생략하고 번호부터 시작한다.
  - 예: `00-overview.md`, `01-설정관리.md`, `todo-01.md`
- 빌드 생성 파일은 배포패키지명 폴더 없이 `/artifacts/` 바로 아래에 둔다.
- 이후 배포 패키지가 둘 이상으로 늘어나면 기본 규칙으로 전환한다.

## 3. 계획 문서와 할 일 목록

### 3.1 파일명 규칙

- 기본 형식은 `<배포패키지 또는 앱 이름>-<구분번호 2자리>-<기능 혹은 용도>.md`이며, 항상 소문자로 작성한다.
- `overview` 문서는 항상 구분번호 `00`을 사용한다.
  - 예: `AIS` 프로그램에 `M`, `CC`, `CL` 단위가 있다면 `M`의 개요 문서는 `m-00-overview.md`
- 내용이 길어 분리가 필요하거나 용도상 분리가 필요한 경우 기본 파일명 뒤에 `-<세부번호>-<제목>`을 붙인다. 세부번호는 `01`부터 시작한다.
  - 예: `m-00-overview-01-패키지구분.md`
- 할 일 목록 파일은 `<배포패키지 또는 앱 이름>-todo-<순번 2자리>.md` 형식을 사용한다 (3.5).
- 배포 패키지가 하나뿐인 경우 2.2에 따라 앱 이름 접두를 생략한다.

### 3.2 문서 헤더와 revision

모든 계획 문서 상단에 다음을 기록한다.

```text
최초 작성일: YYYY-MM-DD
최종 변경일: YYYY-MM-DD
revision: 1
```

- 신규 문서의 `revision`은 `1`로 시작하고, 수정할 때마다 정확히 `1` 증가시키며 `최종 변경일`을 갱신한다. `최초 작성일`은 처음 만든 날짜로 유지한다.
- 할 일 목록 파일은 항목 체크가 빈번하므로 `revision` 없이 `최초 작성일`과 `최종 변경일`만 기록한다.

### 3.3 overview 문서와 phase 정보

- `overview`(구분번호 `00`) 문서에는 개발 순서 등의 phase 정보를 반드시 포함한다.
- 항목 추가, 우선순위 변경, 개발 완료 시 해당 변경을 즉시 phase 정보에 반영하여, phase 정보가 항상 실제 개발 상태와 일치하게 유지한다.

### 3.4 계획용 이미지와 참조 연결

- 계획에서 사용하는 UI 참고 이미지, 아이콘 등 이미지 자산은 2.1의 하위 폴더 규칙에 따라 주인 md 파일과 같은 이름의 폴더에 보관하고, 계획 문서에서는 저장소 상대 경로로 연결한다.
- 이미지가 변경되면 현재 사용하는 이미지와 필요한 원본만 남기고 이전·미사용 이미지는 제거한다.
- 원본을 아이콘 등으로 변형해 사용한 경우 원본과 변형 결과를 모두 보관하고, 이를 사용하는 계획 문서에 둘의 관계를 확인할 수 있는 참조를 연결한다.

### 3.5 할 일 목록(todo)

계획과 구현 사이의 연결 고리다. Claude가 계획에서 할 일을 추출하고, Codex는 계획 문서 전체가 아니라 할 일 목록을 기준으로 구현한다.

#### 파일 운영

- 파일명 순번은 `01`부터 시작하며, 순번이 가장 큰 파일이 "최신 할 일 파일"이다.
- 최신 할 일 파일이 **1000줄을 초과하면** 다음 순번 파일을 생성한다. 이때 미완료 항목을 모두 새 파일로 옮기고, 이전 파일에는 완료·취소 항목만 남겨 이력 보관용으로 동결한다.
- 이에 따라 미완료 할 일은 항상 최신 파일 하나에만 존재하며, Codex는 최신 파일만 확인하면 된다.

#### 할 일 추출 (Claude)

- 계획 문서를 추가·수정할 때마다 실행 가능한 구현 작업을 할 일 항목으로 추출하여 최신 할 일 파일 끝에 추가한다. 할 일 추가 없이 계획만 고치면 계획 작업이 완료된 것이 아니다.
- 항목은 Codex가 계획 전체를 다시 읽지 않아도 착수할 수 있을 만큼 구체적으로 작성하고, 근거 계획 문서와 절 번호를 표기한다.

```text
- [ ] <구체적인 작업 내용> (근거: <계획 문서 파일명> §<절 번호>)
```

- 계획 변경으로 무효화된 할 일은 삭제하지 않고 취소선과 사유를 남긴다.
  - 예: `- [x] ~~기존 작업 내용~~ (취소: 계획 변경, m-01-xxx.md rev 3)`

#### 할 일 수행 (Codex)

- 최신 할 일 파일의 미완료 항목(`- [ ]`)을 기준으로 작업하고, 완료 즉시 항목별로 `- [x]`와 완료일을 기록한다. 여러 항목을 한 번에 작업해도 각각 체크한다.
  - 예: `- [x] <작업 내용> (근거: ...) (완료: 2026-07-20)`
- 사용자가 할 일 목록에 없는 작업을 직접 지시한 경우, 완료 후 해당 작업을 최신 할 일 파일에 완료 상태로 추가하여 기록을 남긴다.

## 4. 작업 수행 규칙

### 4.1 작업 진행 안내

- 지금 무엇을 하고 있는지, 다음에 무엇을 할 것인지를 사용자에게 이야기하면서 진행한다. 설명 없이 결과만 제시하지 않는다.
- 여러 단계 작업은 시작 시 진행 순서를 알리고, 단계가 바뀔 때마다 현재 단계를 알린다. 원래 계획과 다른 방식이 필요해지면 이유를 알린 뒤 진행한다.
- 이 규칙은 Codex와 Claude 모두에게 적용된다.

### 4.2 작업 완료 조건 〔변경 기록 필수 — 누락 반복 금지〕

**변경사항 기록 누락은 이 저장소에서 실제로 반복된 문제이며, 어떤 이유로도 허용하지 않는다. 코드가 완벽해도 기록이 빠지면 그 작업은 미완료다.** 아래 항목이 모두 충족되기 전에는 완료로 보고하지 않으며, 매 작업 종료 시 스스로 점검한다.

1. 요청된 코드 또는 문서 변경이 끝났다.
2. 변경 내용이 관련 계획 문서에 반영되었다.
   - 기능·요구사항의 추가·수정·삭제는 **같은 작업 안에서 즉시** 가장 연관성이 높은 계획 문서에 기록한다. "나중에 반영"은 없다.
   - 기존 계획과 연관성이 낮은 요구사항은 억지로 끼워 넣지 않고 3.1의 규칙에 따라 신규 계획 문서로 작성하며, 해당 단위의 overview phase 정보에 추가한다.
   - 반영 시 3.2에 따라 `revision`과 `최종 변경일`을 갱신하고, 개발 순서에 영향이 있으면 3.3의 phase 정보도 갱신한다.
3. 할 일 목록이 갱신되었다 (3.5). Claude는 계획 변경분의 할 일 추출을, Codex는 수행 항목의 완료 체크를 마쳤다.

- 완료 보고 시에는 갱신한 계획 문서와 할 일 파일의 이름을 함께 명시하여, 사용자가 기록 반영 여부를 즉시 확인할 수 있게 한다.

### 4.3 명시적 요청이 필요한 작업과 자동 수행 작업

- **빌드 체크, 커밋, 푸시, 배포파일 생성**은 각각 사용자의 명시적인 요구가 있을 때만 수행한다. 기능 완료, 빌드 성공, 도구 설치 등 어떤 이유로도 이 작업들을 자동으로 이어서 수행하지 않는다. Claude의 코드 검증 역할에도 동일하게 적용된다.
- 개발, 빌드, 배포패키지 생성에 필요한 프로그램과 패키지의 설치는 사용자의 요청 없이 자동으로 수행한다. 자동 수행은 도구와 의존성 준비까지만 해당한다.

### 4.4 빌드 체크 수행 절차

사용자가 빌드 체크를 요청한 경우:

- 발견된 컴파일 오류와 빌드 실패 원인은 자동으로 수정하고, 빌드가 성공할 때까지 수정과 재빌드를 반복한다. 반복은 최대 5회까지만 수행하며, 5회 안에 성공하지 못하면 중단하고 남은 오류와 수정 내용을 보고한 뒤 지시를 기다린다.
- 자동 수정은 빌드 성공을 위한 최소한의 수정으로 한정하고, 요청하지 않은 기능 추가나 리팩터링을 함께 수행하지 않는다.
- 자동으로 수정한 내용은 파일과 원인 단위로 보고한다.

## 5. 코드 작성 기본 원칙

- 요청된 범위만 구현하며, 요청하지 않은 기능을 임의로 추가하지 않는다.
- 프로젝트에 이미 사용 중인 언어, 프레임워크, 명명 규칙과 코드 스타일을 따른다.
- 하나의 함수와 클래스는 하나의 책임만 가지도록 유지한다.
- 매직 넘버와 매직 문자열을 사용하지 않고 이름 있는 상수 또는 설정값으로 분리한다.
- 사용하지 않는 코드, 주석 처리된 코드, 임시 디버그 출력은 완료 전에 제거한다.
- 동일한 로직을 복사하지 않고 공통 함수 또는 공통 모듈로 분리한다.
- 개별 코드 파일의 길이는 500줄 이내를 목표로 하고, 1000줄을 상한으로 한다. 상한을 초과할 것 같으면 책임 단위로 파일을 분리한다. 자동 생성 코드와 데이터 테이블성 파일은 예외로 한다.

## 6. 설정과 경로

- 설정값을 소스 코드에 하드코딩하지 않고 설정 파일 또는 환경 변수로 분리한다.
- 설정에는 안전한 기본값을 두고, 필수 설정이 없으면 명확한 오류 메시지와 함께 실행을 중단한다.
- 파일 경로는 현재 작업 디렉터리를 가정하지 않고 실행 파일 위치 또는 명시적 기준 경로에서 계산한다. 윈도우즈 서비스는 작업 디렉터리가 `C:\Windows\System32`가 되는 점을 특히 주의한다.
- 윈도우즈 설치형 앱의 표준 경로는 다음과 같다.
  - **설치(실행 파일·리소스)**: `C:\Program Files\DEEPAi\<앱이름>`. 설치는 항상 모든 사용자용(per-machine)으로 수행하며 인스톨러는 관리자 권한을 요구한다. 이 폴더는 읽기 전용으로 취급하고 앱은 런타임에 여기에 쓰지 않는다.
  - **머신 공용 설정·데이터·로그**(모든 사용자가 공유): `%ProgramData%\DEEPAi\<앱이름>`. 윈도우즈 서비스의 설정과 로그도 여기에 둔다.
  - **사용자별 설정·데이터**: `%AppData%\DEEPAi\<앱이름>`. 로밍이 필요 없는 대용량 데이터는 `%LocalAppData%\DEEPAi\<앱이름>`을 사용한다.
- 인스톨러는 `%ProgramData%\DEEPAi\<앱이름>` 폴더를 생성할 때 **Users 그룹에 수정(Modify) 권한을 반드시 부여한다** (예: Inno Setup `[Dirs]` 섹션 `Permissions: users-modify`). 부여하지 않으면 관리자 권한으로 생성된 파일을 일반 사용자로 실행된 앱이 수정하지 못한다.
- `%AppData%\DEEPAi\<앱이름>` 폴더는 인스톨러가 아니라 앱이 첫 실행 시 생성한다. 사용자 계정마다 존재해야 하므로 인스톨러가 대신 만들 수 없다.
- 언인스톨 시 설치 폴더는 제거하되, `%ProgramData%`와 `%AppData%`의 설정·데이터는 기본적으로 남기고 삭제 여부를 사용자에게 확인한다.
- 경로 결합은 문자열 연결이 아닌 언어의 경로 결합 API를 사용한다.

## 7. 문자 인코딩과 시간 처리

- 모든 텍스트 파일 입출력은 UTF-8을 명시하여 수행하며 시스템 기본 코드 페이지에 의존하지 않는다. PowerShell에서도 동일하며, `Get-Content` 사용 시 `-Encoding utf8`을 지정한다.
- 날짜, 숫자, 통화의 파싱과 포매팅은 문화권(Culture)을 명시한다. 저장·통신용 데이터는 문화권 불변 형식을 사용한다.
- 내부 처리와 저장에는 UTC를 사용하고, 사용자에게 표시할 때만 로컬 시각으로 변환한다.
- 저장·교환용 날짜 형식은 `YYYY-MM-DD` 또는 ISO 8601을 사용한다.

## 8. 로깅과 오류 처리

- 예외를 빈 catch로 무시하지 않는다. 처리하지 못하는 예외는 로그를 남기고 상위로 전달한다.
- 오류 메시지에는 원인, 대상(파일명·설정 키·요청 정보 등), 다음 조치를 판단할 수 있는 문맥을 포함한다.
- 로그는 Debug, Info, Warning, Error 수준을 구분해 기록하고, 파일 로그는 크기 또는 기간 기준으로 롤링한다.
- 프로그램의 시작, 정상 종료, 비정상 종료는 반드시 로그에 남긴다. UI가 없는 윈도우즈 서비스와 단독형 프로그램은 로그가 유일한 진단 수단이므로 특히 중요하다.
- 사용자에게 보여주는 오류 메시지와 로그에 남기는 상세 오류를 구분하고, 내부 정보(스택, 경로, 비밀값)는 사용자 메시지에 노출하지 않는다.

## 9. 보안 기본 지침

- 비밀번호, API 키, 연결 문자열의 비밀값을 소스 코드와 저장소에 포함하지 않는다.
- 외부에서 들어오는 모든 입력(파일, 인자, 네트워크 요청, 사용자 입력)은 사용 전에 검증한다.
- 필요 이상의 권한을 요구하지 않는다. 관리자 권한이 필요한 경우 그 이유를 계획 문서에 기록한다.
- 네트워크 통신이 있는 경우 평문 전송을 피하고 TLS 사용을 기본으로 한다.

## 10. 애플리케이션 유형별 기본 지침

- 윈도우즈 앱
  - 설치형 앱의 설치·설정·데이터 경로는 6장의 DEEPAi 표준 경로 규칙을 따른다.
  - UI 스레드에서 장시간 작업을 수행하지 않고 비동기 또는 백그라운드로 처리한다.
  - 종료 시 파일 핸들, 연결, 타이머 등 리소스를 정리한다.
  - 백그라운드 작업의 예외가 앱을 침묵 종료시키지 않도록 전역 예외 처리를 둔다.
- 윈도우즈 서비스
  - 무인 실행을 전제로 하며 대화상자 등 사용자 입력을 요구하는 코드를 두지 않는다.
  - 시작 실패 원인은 이벤트 로그 또는 파일 로그에 남긴다.
  - 서비스 중지 요청 시 진행 중인 작업을 정리하고 제한 시간 안에 종료한다.
- 웹 서비스
  - 수신 포트와 바인딩 주소는 설정으로 분리한다.
  - 상태 점검용 엔드포인트를 제공한다.
  - 요청 처리 실패가 프로세스 전체를 중단시키지 않도록 요청 단위로 오류를 격리한다.
- 단독형 프로그램
  - 성공은 종료 코드 `0`, 실패는 `0`이 아닌 종료 코드로 반환한다.
  - 정상 출력은 표준 출력, 오류는 표준 오류로 구분해 출력한다.
  - 사용자 입력 대기 없이 인자와 설정만으로 실행이 완료될 수 있게 한다.

## 11. 테스트와 검증

- 코드 검증은 정적 검토(로직, 경계 조건, 오류 처리, 지침 준수 확인)를 기본으로 한다. 빌드와 실행을 통한 확인은 4.3에 따라 사용자의 명시적 요청이 있을 때만 수행한다.
- 버그를 수정할 때는 재현 조건과 수정 근거를 함께 기록한다.
- 오류 경로(설정 누락, 파일 부재, 권한 부족, 잘못된 입력)도 정상 경로와 같은 수준으로 검토한다.

## 12. 버전과 build 번호

- 버전 표기는 `vA.B.C build D` 형식을 사용한다. 예: `v1.2.3 build 4`.
- `A`, `B`, `C`는 사용자가 명시적으로 요구한 경우에만 변경한다.
- 첫 번째로 성공한 배포파일의 build 번호는 `1`이다. 사용자가 배포파일 생성을 요구하면, 성공한 build가 없을 때는 후보 build를 `1`로, 있을 때는 정확히 `1` 증가시킨 값으로 설정한 뒤 생성한다.
- 배포파일 생성이 성공한 경우에만 후보 build 번호를 확정하고, 실패하면 시도 전의 번호로 되돌린다. 첫 시도였다면 성공 build가 없는 상태로 되돌린다.
- 일반 코드 수정, 계획 수정, 빌드 체크, 커밋 또는 푸시만으로 build 번호를 변경하지 않는다.
