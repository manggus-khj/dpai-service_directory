# 저장소 공통 작업 지침

```text
최초 작성일: 2026-07-19
최종 변경일: 2026-07-19
revision: 4
```

## 1. 문서의 성격과 역할 분담

### 1.1 문서 체계

- 이 파일(`AGENTS.md`)은 저장소의 표준 개발 지침이며, 개별 애플리케이션 작업을 이유로 수정하지 않는다.
- 이 파일에는 저장소 전체에 적용되는 일반 지침과 참조 연결 규칙만 기록한다. 개별 애플리케이션이나 배포 패키지의 요구사항, 설계, 일정, 구현 단계는 기록하지 않는다.
- 해당 앱의 기본적인 명칭과 개발 단위는 `README.md`를 수정하여 지정한다. `README.md`에는 계획 문서들의 용도, 위치, 파일명을 지정한다.
- 저장소의 공통 지침 원본은 `AGENTS.md` 하나로 유지한다. `CLAUDE.md`에는 세부 지침을 복제하지 않고 `AGENTS.md`를 참조하라는 내용만 둔다.
- 이 파일 상단에도 계획 문서와 같은 형식의 `최초 작성일`, `최종 변경일`, `revision` 헤더를 두며, 이 파일을 수정할 때마다 `revision`을 정확히 `1` 증가시키고 `최종 변경일`을 갱신한다.
- 공통 지침 자체를 변경해야 하는 경우에도 `AGENTS.md`만 수정하며, `CLAUDE.md`의 참조 연결은 그대로 유지한다.

### 1.2 기본 역할

- Codex는 개발과 코드 수정을 기본 담당으로 한다.
- Claude는 계획 작성과 코드 검증을 기본 담당으로 한다.

## 2. 저장소 구조와 위치 규칙

저장소의 표준 최상위 구조는 다음과 같다. 아래 규칙 외의 위치에 같은 성격의 파일을 두지 않는다.

| 위치 | 용도 |
|---|---|
| `/docs/plan/` | 모든 계획 문서 (하위 폴더 없이 파일명 규칙으로 구분) |
| `/src/<배포패키지명>/` | 코드 소스 |
| `/common/` | 빌드와 인스톨러 생성을 위한 공용 패키지 |
| `/artifacts/<배포패키지명>/` | 빌드 과정에서 생성되는 파일 |
| `/installer/` | 최종 설치패키지 (하위 폴더 없음) |

### 2.1 계획 문서 위치

- 모든 계획 문서의 기준 루트는 저장소 루트의 `/docs/plan/`이다.
- 배포 패키지나 애플리케이션별 하위 폴더를 만들지 않고 `/docs/plan/` 바로 아래에 둔다. 배포 패키지와 애플리케이션의 구분은 폴더가 아니라 3장의 파일명 규칙으로 한다.
- `/<배포패키지명>/docs/plan/`처럼 `/docs/plan/` 이외의 위치에 계획 루트를 만들지 않는다.
- 여러 패키지에 공통으로 적용되는 프로젝트 계획과 공통 결정도 `/docs/plan/` 바로 아래에 둔다.
- `/docs/plan/` 내부에 하위 폴더를 만드는 경우는 md가 아닌 다른 종류의 파일(이미지 등)을 보관할 때로 한정하며, 그 파일들을 사용하는 주인 md 파일과 같은 이름의 폴더를 만들어 그 안에 넣는다.

### 2.2 코드 소스 위치

- 코드 소스는 반드시 저장소 루트 기준 `/src/<배포패키지명>/` 아래에 둔다.
- 여러 배포 패키지의 소스를 하나의 패키지 폴더에 섞어 두지 않는다.

### 2.3 빌드와 인스톨러 생성용 패키지 위치

- 빌드와 인스톨러 생성을 위한 패키지(빌드 도구, 인스톨러 제작 도구, 공용 컴포넌트, 재배포 가능한 런타임 등)는 `/src/` 아래에 두지 않는다.
- 이 패키지들은 저장소 루트 기준 `/common/` 아래에 두고, 모든 배포 패키지가 이를 공용으로 사용한다.
- 하나의 프로젝트에서 여러 배포 패키지를 생성하더라도 같은 컴포넌트를 배포 패키지마다 중복 보관하지 않고 `/common/`의 것 하나만 사용한다.
- 배포 패키지가 하나뿐인 경우에도 이 패키지들을 개별 패키지 폴더 등 다른 위치에 두지 않는다. 위치가 일정하지 않으면 존재 여부를 파악하기 어렵기 때문이다.
- 특정 배포 패키지 전용의 빌드·인스톨러 설정 파일이나 아이콘 등의 자산은 이 규칙의 대상이 아니며, 기존 위치 규칙을 그대로 따른다.

### 2.4 배포 산출물 위치

- 설치패키지(인스톨러)는 저장소 루트 기준 `/installer/` 바로 아래에 생성하며, 그 아래에 배포 패키지별 하위 폴더를 만들지 않는다.
- 빌드 과정에서 생성되는 파일은 저장소 루트 기준 `/artifacts/<배포패키지명>/` 아래에 위치시킨다.
- 여러 배포 패키지의 빌드 생성 파일을 하나의 패키지 폴더에 섞어 두지 않는다.
- `/artifacts/` 아래의 빌드 중간 파일을 최종 설치패키지로 취급하지 않는다.

## 3. 계획 문서 작성 규칙

### 3.1 파일명 규칙

- 기본 형식은 `<배포패키지 또는 앱 이름>-<구분번호 2자리>-<기능 혹은 용도>.md`이다.
- 파일명은 항상 소문자로 작성한다.
- 구분번호는 두 자리 숫자로 표기하며, `overview` 문서는 항상 구분번호 `00`을 사용한다.
  - 예: `AIS` 프로그램에 `M`, `CC`, `CL` 단위가 있다면 `M`의 개요 문서는 `m-00-overview.md`가 된다.
- 파일의 내용이 너무 길어져 세부적으로 분리해야 하거나 용도상 분리가 필요한 경우, 기본 파일명 뒤에 `-<세부번호>-<제목>`을 붙여 분리한다.
  - 세부번호는 항상 `01`부터 시작한다.
  - 예: `m-00-overview-01-패키지구분.md`

### 3.2 문서 헤더와 revision

모든 계획 문서 상단에는 다음 항목을 기록한다.

```text
최초 작성일: YYYY-MM-DD
최종 변경일: YYYY-MM-DD
revision: 1
```

- 새 계획 문서의 `revision`은 `1`로 시작한다.
- 계획 문서를 수정할 때마다 `revision`을 정확히 `1` 증가시키고 `최종 변경일`을 갱신한다.
- `최초 작성일`은 문서를 처음 만든 날짜로 유지한다.

### 3.3 overview 문서와 phase 정보

- `overview`(구분번호 `00`) 문서에는 개발 순서 등의 phase 정보를 반드시 포함한다.
- 항목이 추가되거나, 우선순위가 바뀌거나, 개발이 완료된 경우에는 해당 변경을 즉시 phase 정보에 반영한다.
- phase 정보는 항상 실제 개발 상태와 일치하도록 유지하여, 개발 시 잘못된 순서나 누락으로 인한 오류가 발생하지 않게 한다.

### 3.4 계획용 이미지와 참조 연결

- 계획에서 사용하는 UI 참고 이미지, 아이콘, 기타 이미지 자산은 2.1의 하위 폴더 규칙에 따라 해당 이미지를 사용하는 주인 md 파일과 같은 이름의 폴더에 보관한다.
- 계획 문서에서는 저장소 상대 경로로 이미지와 관련 문서를 연결한다.
- 이미지가 변경되면 현재 계획에서 실제로 사용하는 이미지와 필요한 원본만 남기고 이전·미사용 이미지는 제거한다.
- 업로드한 원본을 아이콘이나 다른 형태로 변형해 사용한 경우 원본과 실제 사용하는 변형 결과를 모두 보관한다.
- 변형 자산을 사용하는 계획 문서에는 원본과 변형 결과의 관계를 확인할 수 있도록 참조를 연결한다.

### 3.5 계획 문서의 최신성 유지

- 새로운 기능이나 요구사항이 추가되거나, 기존 기능·요구사항이 수정 또는 삭제되면 해당 변경을 반드시 관련 계획 문서에 즉시 반영한다.
- 추가·변경·삭제된 요구사항은 현재의 계획 문서 중 가장 연관성이 높은 문서의 해당 위치에 기록한다.
- 기존 계획과 연관성이 낮은 요구사항은 기존 문서에 억지로 끼워 넣지 않고, 3.1의 파일명 규칙에 따라 신규 계획 문서로 작성한다.
- 코드만 변경하고 계획 문서를 갱신하지 않은 상태로 작업을 완료 처리하지 않는다.
- 반영 시에는 3.2의 규칙에 따라 `revision`과 `최종 변경일`을 갱신하고, 개발 순서에 영향이 있으면 3.3의 phase 정보도 함께 갱신한다. 신규 계획 문서를 작성한 경우에도 해당 단위의 overview 문서 phase 정보에 추가한다.

## 4. 코드 작성 기본 원칙

- 요청된 범위만 구현하며, 요청하지 않은 기능을 임의로 추가하지 않는다.
- 프로젝트에 이미 사용 중인 언어, 프레임워크, 명명 규칙과 코드 스타일을 따른다.
- 하나의 함수와 클래스는 하나의 책임만 가지도록 유지한다.
- 매직 넘버와 매직 문자열을 사용하지 않고 이름 있는 상수 또는 설정값으로 분리한다.
- 사용하지 않는 코드, 주석 처리된 코드, 임시 디버그 출력은 완료 전에 제거한다.
- 동일한 로직을 복사하지 않고 공통 함수 또는 공통 모듈로 분리한다.
- 개별 코드 파일의 길이는 500줄 이내를 목표로 하고, 1000줄을 상한으로 한다. 상한을 초과할 것 같으면 책임 단위로 파일을 분리한다. 자동 생성 코드와 데이터 테이블성 파일은 예외로 한다.

## 5. 설정과 경로

- 설정값을 소스 코드에 하드코딩하지 않고 설정 파일 또는 환경 변수로 분리한다.
- 설정에는 안전한 기본값을 두고, 필수 설정이 없으면 명확한 오류 메시지와 함께 실행을 중단한다.
- 파일 경로는 현재 작업 디렉터리를 가정하지 않고 실행 파일 위치 또는 명시적 기준 경로에서 계산한다. 윈도우즈 서비스는 작업 디렉터리가 `C:\Windows\System32`가 되는 점을 특히 주의한다.
- 프로그램이 생성하는 데이터와 로그는 설치 폴더가 아닌 표준 위치(`%ProgramData%`, `%AppData%` 또는 설정으로 지정한 경로)에 저장한다.
- 경로 결합은 문자열 연결이 아닌 언어의 경로 결합 API를 사용한다.

## 6. 문자 인코딩과 시간 처리

- 모든 텍스트 파일 입출력은 UTF-8을 명시하여 수행하며 시스템 기본 코드 페이지에 의존하지 않는다.
- PowerShell에서 텍스트 파일을 읽을 때는 항상 UTF-8을 명시한다. `Get-Content`를 사용할 때는 `-Encoding utf8`을 지정한다.
- PowerShell 스크립트에서도 시스템 기본 코드 페이지에 의존하지 않고 텍스트 입력을 UTF-8로 로딩한다.
- 날짜, 숫자, 통화의 파싱과 포매팅은 문화권(Culture)을 명시한다. 저장·통신용 데이터는 문화권 불변 형식을 사용한다.
- 내부 처리와 저장에는 UTC를 사용하고, 사용자에게 표시할 때만 로컬 시각으로 변환한다.
- 저장·교환용 날짜 형식은 `YYYY-MM-DD` 또는 ISO 8601을 사용한다.

## 7. 로깅과 오류 처리

- 예외를 빈 catch로 무시하지 않는다. 처리하지 못하는 예외는 로그를 남기고 상위로 전달한다.
- 오류 메시지에는 원인, 대상(파일명·설정 키·요청 정보 등), 다음 조치를 판단할 수 있는 문맥을 포함한다.
- 로그는 Debug, Info, Warning, Error 수준을 구분해 기록한다.
- 파일 로그는 크기 또는 기간 기준으로 롤링하여 무한히 커지지 않게 한다.
- 프로그램의 시작, 정상 종료, 비정상 종료는 반드시 로그에 남긴다. UI가 없는 윈도우즈 서비스와 단독형 프로그램은 로그가 유일한 진단 수단이므로 특히 준수한다.
- 사용자에게 보여주는 오류 메시지와 로그에 남기는 상세 오류를 구분하고, 내부 정보(스택, 경로, 비밀값)는 사용자 메시지에 노출하지 않는다.

## 8. 보안 기본 지침

- 비밀번호, API 키, 연결 문자열의 비밀값을 소스 코드와 저장소에 포함하지 않는다.
- 외부에서 들어오는 모든 입력(파일, 인자, 네트워크 요청, 사용자 입력)은 사용 전에 검증한다.
- 필요 이상의 권한을 요구하지 않는다. 관리자 권한이 필요한 경우 그 이유를 계획 문서에 기록한다.
- 네트워크 통신이 있는 경우 평문 전송을 피하고 TLS 사용을 기본으로 한다.

## 9. 애플리케이션 유형별 기본 지침

공통 지침에 더해 유형별로 다음을 지킨다.

- 윈도우즈 앱
  - UI 스레드에서 장시간 작업을 수행하지 않고 비동기 또는 백그라운드로 처리한다.
  - 종료 시 파일 핸들, 연결, 타이머 등 리소스를 정리한다.
  - 백그라운드 작업의 예외가 앱을 침묵 종료시키지 않도록 전역 예외 처리를 둔다.
- 윈도우즈 서비스
  - 무인 실행을 전제로 하며 대화상자 등 사용자 입력을 요구하는 코드를 두지 않는다.
  - 시작 실패 원인은 이벤트 로그 또는 파일 로그에 남긴다.
  - 서비스 중지 요청 시 진행 중인 작업을 정리하고 제한 시간 안에 종료한다.
- 웹 서비스
  - 수신 포트와 바인딩 주소는 설정으로 분리한다.
  - 상태 점검용 엔드포인트를 제공한다.
  - 요청 처리 실패가 프로세스 전체를 중단시키지 않도록 요청 단위로 오류를 격리한다.
- 단독형 프로그램
  - 성공은 종료 코드 `0`, 실패는 `0`이 아닌 종료 코드로 반환한다.
  - 정상 출력은 표준 출력, 오류는 표준 오류로 구분해 출력한다.
  - 사용자 입력 대기 없이 인자와 설정만으로 실행이 완료될 수 있게 한다.

## 10. 테스트와 검증

- 코드 검증은 정적 검토(로직, 경계 조건, 오류 처리, 지침 준수 확인)를 기본으로 한다.
- 빌드와 실행을 통한 확인은 11장에 따라 사용자의 명시적 요청이 있을 때만 수행한다.
- 버그를 수정할 때는 재현 조건과 수정 근거를 함께 기록한다.
- 오류 경로(설정 누락, 파일 부재, 권한 부족, 잘못된 입력)도 정상 경로와 같은 수준으로 검토한다.

## 11. 작업 수행 규칙

### 11.1 작업 진행 안내

- 작업을 진행할 때는 지금 무엇을 하고 있는지, 다음에 무엇을 할 것인지를 사용자에게 이야기하면서 진행한다. 설명 없이 조용히 작업만 수행하고 결과만 제시하지 않는다.
- 여러 단계로 이루어진 작업은 시작할 때 진행 순서를 간단히 알리고, 단계가 바뀔 때마다 현재 어느 단계인지 알린다.
- 작업 중 원래 계획과 다른 방식이 필요해지면 그 이유를 알린 뒤 진행한다.
- 이 규칙은 Codex와 Claude 모두에게 적용된다.

### 11.2 명시적 요청이 필요한 작업

기능 추가나 버그 수정이 끝났다는 이유만으로 다음 작업을 자동 수행하지 않는다.

- 빌드 체크
- 커밋
- 푸시
- 배포파일 생성

위 작업은 각각 사용자의 명시적인 요구가 있을 때만 수행한다. Claude의 코드 검증 역할도 사용자의 요청이 없는 빌드 체크, 커밋, 푸시 또는 배포파일 생성을 허용하지 않는다.

### 11.3 자동으로 수행하는 작업

- 개발, 빌드, 배포패키지 생성에 필요한 프로그램과 패키지는 사용자의 요청 없이 자동으로 설치한다.
- 자동 설치는 도구와 의존성 준비까지만 해당하며, 설치했다는 이유로 빌드 체크, 커밋, 푸시, 배포파일 생성을 이어서 수행하지 않는다.

### 11.4 빌드 체크 수행 절차

사용자가 빌드 체크를 요청한 경우에는 다음을 따른다.

- 빌드 체크에서 발견된 컴파일 오류와 빌드 실패 원인은 별도의 요청 없이 자동으로 수정하고, 빌드가 성공할 때까지 수정과 빌드 체크를 반복한다.
- 수정과 재빌드의 반복은 최대 5회까지만 수행한다. 5회 안에 빌드가 성공하지 못하면 자동 수정을 중단하고, 남은 오류와 그때까지의 수정 내용을 사용자에게 보고한 뒤 지시를 기다린다.
- 자동 수정의 범위는 빌드를 성공시키기 위한 최소한의 수정으로 한정하며, 요청하지 않은 기능 추가나 리팩터링을 함께 수행하지 않는다.
- 자동으로 수정한 내용은 파일과 원인 단위로 사용자에게 보고한다.
- 빌드가 성공했다는 이유로 커밋, 푸시, 배포파일 생성을 이어서 수행하지 않는다.

## 12. 버전과 build 번호

- 버전 표기는 `vA.B.C build D` 형식을 사용한다. 예: `v1.2.3 build 4`.
- `A`, `B`, `C`는 사용자가 명시적으로 요구한 경우에만 변경한다.
- 첫 번째로 성공한 배포파일의 build 번호는 `1`이다.
- 사용자가 배포파일 생성을 요구하면 아직 성공한 build가 없을 때는 후보 build를 `1`로, 기존 성공 build가 있을 때는 후보 build를 정확히 `1` 증가시킨 값으로 설정한 뒤 배포파일을 생성한다.
- 배포파일 생성이 성공한 경우에만 후보 build 번호를 확정한다.
- 배포파일 생성이 실패하면 시도 전에 사용하던 build 번호로 되돌린다. 첫 배포 시도였다면 성공 build가 없는 상태로 되돌린다.
- 일반 코드 수정, 계획 수정, 빌드 체크, 커밋 또는 푸시만으로 build 번호를 변경하지 않는다.
